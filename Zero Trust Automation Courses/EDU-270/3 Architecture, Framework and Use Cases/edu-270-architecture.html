Architecture, Framework & Use Cases

Zscaler API architecture, the Zero Trust Automation Framework, traditional vs. new approaches, and real-world use cases for ZIA, ZPA, and Client Connector.

The framework is the foundation of automation.

Understanding how the Zscaler Zero Trust Automation architecture works — and seeing it applied to real use cases — transforms abstract concepts into actionable skills.

💭THINK FIRST · API ARCHITECTURE

Every API request travels through multiple layers before reaching the actual data. The OneAPI architecture has a Central Authority that handles authentication, then routes to specific capabilities, then enforces security controls. Think about why each layer exists as a separate concern — and what would break if layers were combined.

  • Why is it better to have a centralized API Gateway handling authentication rather than each backend service (ZIA, ZPA, ZDX) independently validating tokens?
  • "Browser activities are driven by the same APIs as programmatic access." What does this tell you about the consistency between what an admin does manually in the portal versus what an automation script does via API?
  • Rate limiting has two tiers: burst limits (lower bound) and volume limits (upper bound). Why are these separate — what specific scenario does the burst limit protect against that the hourly volume limit does not?
Q1 — MODEL ANSWER
A centralized API Gateway concentrates authentication logic in one place, which means token validation, key rotation, and revocation are consistent across every product. If each backend (ZIA, ZPA, ZDX) validated tokens independently, you'd risk divergent implementations, inconsistent enforcement of policies, and far more attack surface. Centralization also enables shared rate limiting, logging, and access policy enforcement — all things that should not be reinvented per service.
Q2 — MODEL ANSWER
It tells you the portal is just a graphical front-end on top of the same APIs your script uses. Anything you can do as an admin in the browser, you can do programmatically — and vice versa. This guarantees parity between manual and automated workflows, so an automation script will never be artificially restricted compared to a human admin, and changes happen through the same audited code path.
Q3 — MODEL ANSWER
The burst limit protects against short-duration spikes — for example, a buggy script in a tight loop firing 1,000 requests in a second. An hourly volume cap wouldn't catch that because the script could exceed instantaneous capacity well before the hour ends, overwhelming the gateway. The burst limit smooths traffic in real time, while the volume limit ensures fair long-term usage. Together they prevent both DoS-style spikes and sustained over-consumption.
API ARCHITECTURE AND ACCESS

Zscaler API Architecture and Access

Zscaler APIs provide a secure and scalable approach to programmatically manage and configure Zscaler services, enabling seamless policy enforcement, configuration management, and comprehensive logging through an authenticated and documented API gateway.

Zscaler API Architecture: Central Authority (API Gateway) → API Capabilities (Policy, Log, Config Management) → Authentication & Security → Developer Support

Central Authority Level

API Gateway
The central hub for applying policies and retrieving data. All API requests are routed through the OneAPI Gateway, which enforces authentication, access policies, and rate limits.
Authentication
The gateway authenticates with the Central Authority, which may include the Admin Portal. ZIdentity serves as the OAuth server for all authentication.
Browser and Programmatic Access
Activities done in the browser are driven by APIs, which can also be used programmatically for various controls. Both methods use the same underlying API infrastructure.

API Capabilities

Policy Management
Make policy changes and configurations programmatically. Update application segments and manage access policies at scale.
Log Management
Stream logs for monitoring and analysis. Integrate with SIEM tools for real-time security visibility.
Configuration Management
Handle SCIM and IdP configurations. Manage custom service edges, trusted networks, and connector groups.

Authentication and Security

Authentication Control
Manage authentication processes. ZIdentity OAuth tokens validated at the API Gateway level.
Rate Limiting
Ensure API usage is controlled and fair. Two-tier system: burst limits (lower bound) and volume limits (upper bound).
Identity and Validation
Manage identity verification and payload validation. JWT signature verified against client's public key or X.509 certificate.
Logging
Comprehensive logging for monitoring and troubleshooting all API interactions.
Third-Party Integrations
Control and integrate with third-party partner APIs through a standardised interface.
Developer Support
All APIs documented and available through the Zscaler Help Portal for easy reference and implementation.
Analogy
The OneAPI architecture is like airport security. The Central Authority (API Gateway) is the single security checkpoint everyone passes through regardless of destination. API Capabilities are the different terminals you reach after clearing security. The Authentication & Security layer is passport control and screening. Developer Support is the airport directory that explains how to navigate. You can't skip any layer — and that's the point.

ASK A QUESTION
MY NOTES
💭THINK FIRST · TRADITIONAL FRAMEWORK

Before OneAPI, the traditional Zscaler automation framework required separate authentication flows for ZIA, ZPA, and Client Connector. Before reading about the problems, try to reason through what an automation script that manages all three products would actually look like — and where things would go wrong.

  • If you need to rotate API credentials for security reasons and you have separate credentials for ZIA, ZPA, and ZCC — what's the operational burden? What's the risk if you forget to rotate one of them?
  • If ZIA uses a different rate limiting scheme than ZPA, what does your automation script need to do differently when calling each product — and how does that complexity compound as you add more products?
  • In a merger/acquisition scenario, a company inherits two Zscaler tenants. Under the traditional framework, how many separate sets of credentials would an admin managing both need to handle simultaneously?
Q1 — MODEL ANSWER
The operational burden is significant: each product has its own admin portal, its own credential storage, and its own rotation procedure, so rotating credentials means doing the same task three times in three different places. If you forget to rotate one, you've left a stale credential that may still be valid — an attacker who compromised that credential keeps access to that product indefinitely. This is exactly the kind of inconsistency that erodes a zero-trust posture.
Q2 — MODEL ANSWER
Your script needs product-specific retry logic, back-off intervals, and header parsing for each product's rate-limit response. Adding a new product means writing yet another adapter and maintaining yet another quirk. The complexity compounds because a single integration change (say, ZIA modifies its rate-limit window) forces re-testing of the entire automation suite, and there's no shared error-handling layer to inherit from.
Q3 — MODEL ANSWER
With three products per tenant and two tenants, the admin manages six independent credential sets simultaneously — plus separate auth flows, endpoints, and rate-limit behaviours per product. That's a tracking nightmare for rotation, audit, and revocation, and it's exactly the scenario where credentials end up reused or sticky in scripts long after they should have been retired.
TRADITIONAL AUTOMATION FRAMEWORK

Traditional Zscaler Automation Framework

Before OneAPI, the traditional automation framework had significant limitations. Each product required separate management, creating complexity and error potential.

Challenges of the Traditional Approach

Deploying new SaaS solutions at scale is complex and prone to errors, leading to slow rollouts and increased costs. Fragmented ownership among multiple administrators — particularly in the context of mergers or acquisitions — results in accountability issues.

ChallengeDescriptionImpact
Credential ManagementCredentials maintained within the administrator console of each productMultiple sets of credentials to manage and rotate per product
API Endpoint FragmentationEach product API has a separate API endpointScripts need multiple endpoints; increased complexity
Common Functionality VariationsVariations on common API features such as Rate LimitingCannot apply uniform rate-limit handling across products
Multiple Auth FlowsEach product required separate registration, auth, and token requestsMultiple sets of code for each product's authentication flow
Traditional framework: customers must separately authenticate and obtain tokens for ZIA (zsapi.zscaler.net), ZPA (config.zpatwo.net), and Client Connector (api-mobile.zscalerbeta.net)
!
THE CORE PROBLEM
Before OneAPI, switching between different API products required repeated registration, authorization, token requests, and token grants. Automation scripts had to manage multiple endpoints and tokens, raising the chances of mistakes and security issues. OneAPI simplifies this by unifying API interactions under a single framework, reducing redundant code and error risk.
Analogy
The traditional framework was like having separate usernames and passwords for every website on the internet — no password manager, no SSO, just a growing list of credentials to manage manually. Inevitably you start reusing passwords, writing them down, or forgetting to revoke them when someone leaves. OneAPI is the equivalent of a password manager and SSO combined — one credential, one place to manage it, one place to revoke it.

ASK A QUESTION
MY NOTES
💭THINK FIRST · ZTA FRAMEWORK

The Zero Trust Automation Framework centralizes everything through ZIdentity + OneAPI. Think about what "everything external that interacts with Zscaler over automation must create an API client" means in practice — including partners and third-party tools, not just your own scripts.

  • Partners and technology partners must create API clients — they can't use admin credentials. What security principle does this enforce, and what specific risk does it prevent compared to sharing admin credentials with a third party?
  • Step-up authentication (AL1–AL4) escalates the required auth strength when accessing sensitive resources. Why is dynamic authentication levels better than just requiring the strongest (AL4 hardware key) for all operations?
  • The ZTA framework flow is: Authenticate → ZIdentity → Token → OneAPI → Product. At which point in this flow could a legitimate security control interrupt the request — and why would you want that capability?
Q1 — MODEL ANSWER
This enforces the principle of least privilege and non-repudiation. An API client has a scoped identity, narrowly defined permissions, and its own audit trail — so partners can only touch what they need, and every action is attributable to them rather than to a shared admin account. Sharing admin credentials would give a third party full god-mode access with no accountability and no easy revocation, which is exactly the anti-pattern zero trust eliminates.
Q2 — MODEL ANSWER
Dynamic levels follow the risk: low-risk reads can use AL1 with no friction, while sensitive operations (like creating an API client or revealing OTPs) can demand AL4. Forcing AL4 universally would make admins carry hardware keys for every trivial action, which leads to workarounds and shared keys — the opposite of secure. Step-up strikes a balance between usability and security by matching authentication strength to the sensitivity of the operation.
Q3 — MODEL ANSWER
The OneAPI Gateway is the natural interception point — it sees the token, the requested operation, and any rate-limit or access-policy state. You'd want it there because it can enforce quota, revoke a compromised token, trigger step-up authentication, or block based on a risk signal before the request ever reaches the product. Centralizing enforcement at the gateway means a single security update protects all products simultaneously.
ZERO TRUST AUTOMATION FRAMEWORK

Zscaler Zero Trust Automation Framework

The ZIdentity framework centralises identity management within the Zscaler ecosystem, integrating authentication, entitlement management, and external identity providers. This represents the modern replacement for the fragmented traditional approach.

The New Model: Everything external that is going to interact with Zscaler over automation — customers, partners, technology partners — must create API clients. All users authenticate via ZIdentity, which is Zscaler's identity platform hosting all registered API clients. ZIdentity is the OAuth Server which issues tokens required to interact with Zscaler products.
Zero Trust Automation Framework: Customers authenticate through ZIdentity (acme.zslogin.net) → Token → OneAPI Gateway (api.zsapi.net) → ZIA, ZPA, Client Connector, Other Products

Framework Components

OneAPI Platform
Common consistent API experience for all types of customers. Single endpoint: api.zsapi.net.
ZIdentity
API Credential Management. OAuth Server that issues tokens. Hosts all registered API clients.
Single Endpoint
A single endpoint to use for all Zscaler APIs. No more managing multiple product-specific URLs.
OneAPI Gateway
Single endpoint supporting common functionality: Token Validation, Rate Limit, Access Policy, Quota, and much more.

Step-Up Authentication

OneAPI supports step-up authentication — a dynamic request for stronger authentication when accessing sensitive resources or when risk signals are detected.

LevelAuthentication Strength
AL1Basic authentication — standard login
AL2MFA with second factor (SMS OTP, email OTP)
AL3Strong MFA (TOTP via Google Authenticator)
AL4Hardware-backed authentication (FIDO key)

Flow: Login → Access Sensitive Resource → Step-Up Triggered → MFA Challenge → Access Granted. Benefits: Enhanced Security, Dynamic Risk-Based Access, Simplified Compliance.

Analogy
The ZTA Framework is like a secure government facility. ZIdentity is the security office that issues clearances and badges — no badge, no entry. The OneAPI Gateway is the main entrance checkpoint that validates your badge and determines which areas you're cleared for. Step-up authentication (AL1–AL4) is like different clearance levels — some rooms require showing your badge, others require a fingerprint, others require a physical hardware key.

ASK A QUESTION
MY NOTES
💭THINK FIRST · USE CASES

Three use cases are demonstrated: block a URL in ZIA, create an App Connector group in ZPA, retrieve device OTPs in ZCC. Before reading the step-by-step details, think about what you'd need to know to execute each one independently — what inputs, what API calls, what verification steps.

  • In the ZIA URL blocking demo, you need to update a URL filtering policy via PUT request. What must you know before you can construct that request — specifically, how would you find the correct category ID to reference in the JSON body?
  • The ZPA App Connector group creation requires longitude, latitude, and location in the request body. Why does a software configuration object need geographic coordinates?
  • Client Connector OTP retrieval returns credentials for all devices in your tenant. What security controls should be in place before you run a script that retrieves sensitive credentials for potentially tens of thousands of devices?
Q1 — MODEL ANSWER
You need to first GET the URL Filtering Policy to inspect the existing categories and find the numeric category ID associated with the rule you want to update (for example, the "Gambling" category). The category ID is what you reference in the JSON body of your PUT request — without it, the API has no way to know which rule object you're modifying. Always read before you write: confirm the current state, then craft an update that targets the right resource.
Q2 — MODEL ANSWER
App Connector groups have geographic attributes because ZPA uses location data to optimize routing — clients should connect to the nearest healthy App Connector to minimize latency. Latitude and longitude let ZPA make intelligent placement and failover decisions, and the location string provides human-readable context for admins. Even though it's "just configuration", that metadata drives real network behaviour.
Q3 — MODEL ANSWER
At minimum: a scoped API client with read-only access to the ZCC endpoints needed, step-up authentication (AL3 or AL4) for the user invoking the script, audit logging of every retrieval, and encrypted storage for the retrieved OTPs (never plain text on disk). You also want rate-limit awareness and the smallest possible blast radius — pull only the devices you need, not the whole tenant — and a clear, short-lived purpose for the credentials so they're rotated immediately after use.
COMMON USE CASES

Common Use Cases for Zscaler Zero Trust Automation

These practical examples demonstrate how the Zero Trust Automation framework and OneAPI are applied to solve real administrative challenges across Zscaler products.

USE CASE 01
Zscaler Internet Access (ZIA)

ZIA API provides capabilities to automate and manage: URL filtering, policy management, user and group configurations, traffic forwarding, and reporting.

Demo scenario: Block a gambling URL using ZIA API. Steps:

  • Verify the URL is accessible in a test browser session
  • Navigate to ZIA Admin Portal → Manage → URL Categories
  • Check if the URL filtering category (e.g., "Gambling") is enabled to block
  • Use Postman REST API client with the ZIA collection
  • Call the URL Filtering Policy endpoint, check if gambling category is blocked
  • Send a PUT request to update the URL filtering policy using the category ID
  • Add the URL to the custom categories in the JSON body
  • Verify the URL is now blocked in the browser

Prerequisite: Add an API key. Only admins with the API Key Management role can create API keys for ZIA.

USE CASE 02
Zscaler Private Access (ZPA)

ZPA API enables automation of: managing application access policies, configuring App Connector groups, retrieving user device information, monitoring system health.

Demo scenario: Create an App Connector group using the OneAPI ZPA collection.

  • Log in to ZPA Admin Portal using ZIdentity credentials
  • Navigate to Configuration → App Connectors section to verify current state
  • Open Postman REST API client, load the ZPA Postman collection
  • All prerequisites are completed: OneAPI client authenticated
  • Find the customer ID from the ZPA Admin Portal → Administration → Company Details
  • The JSON body includes name, longitude, latitude, and location for the new connector group
  • Send the POST request — expected response: 201 Created
  • Verify the new App Connector group appears in the ZPA Admin Portal
USE CASE 03
Zscaler Client Connector

Zscaler Client Connector API provides functionalities to automate the deployment, configuration, and monitoring of the Zscaler Client Connector across user devices.

Demo scenario: Use the Client Connector API to retrieve all devices registered in the tenant and get device OTP credentials.

  • The ZCCBase variable in Postman is set to the Client Connector base URL (api.zsapi.net/zcc/papi/public/v1)
  • Client ID and Client Secret loaded in the Authorization section
  • Call the v1/getDevices endpoint
  • The API returns all devices registered in the tenant
  • For each device, retrieve the one-time passwords (OTPs)
  • The response iterates through each device's details and OTP credentials
i
POSTMAN COLLECTION
For all use cases, Zscaler provides a Postman collection on help.zscaler.com. The collection includes preconfigured requests, payloads, and environment variables. Download the most recent version of the collection for each product before starting your automation work.
Analogy
These three use cases are test drives for the API. ZIA URL blocking = changing a traffic signal remotely via API (policy update happens in the cloud immediately). ZPA App Connector group creation = opening a new service entrance in a corporate building by sending an instruction — the building reconfigures itself. ZCC device OTP retrieval = requesting the master key list for all doors through a secure authenticated portal, rather than physically visiting each lock.
