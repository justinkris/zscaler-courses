Study Guide — Introduction · EDU-270

Introduction to APIs

API fundamentals, RESTful principles, HTTP methods, request/response cycle, and why APIs power Zscaler Zero Trust Automation.

APIs are the language of automation.

Before you can automate Zero Trust, you need to understand the building blocks — what an API is, how it works, and why Zscaler built OneAPI to unify them all.

💭THINK FIRST · COURSE OVERVIEW

You're about to learn how to automate security policies across an enterprise. Before diving in, think about what "automation" really means in an IT security context — and what the risks are when the same script can change a policy across 1,000 sites simultaneously.

  • If a script could push a policy change to every branch site in seconds, what happens if that policy has an error — and how is that different from making one manual change at a time?
  • What's the difference between a tool that helps you make changes faster versus one that helps you make changes correctly and consistently?
  • Why do you think APIs need to follow standards (like REST) rather than each vendor building their own bespoke interface — what would integration look like without standards?
Q1 — MODEL ANSWER
The "blast radius" of an automated change is fundamentally different from a manual one — a single bad script can corrupt policy on 1,000 sites in seconds, whereas manual changes have natural friction that lets you catch errors after the first or second site. This is why production automation must include staged rollouts, dry-run modes, and rollback plans; the speed advantage of APIs is also the speed at which mistakes propagate. Treat every script run as a deployment event, not a routine config tweak.
Q2 — MODEL ANSWER
Speed and correctness are different problems. A tool that's just faster can replicate a flawed configuration 100 times; a tool that enforces consistency uses templates, version control, and validation to make sure every site ends up with the same intended state. APIs combined with JSON templates and CI/CD pipelines lean toward the second kind — they're not just typing faster, they're encoding intent so the result is deterministic.
Q3 — MODEL ANSWER
Without standards like REST, every vendor would invent custom verbs, custom data formats, and custom auth schemes, forcing developers to relearn fundamentals for each product — integration would resemble the pre-USB world where every device needed its own cable. REST gives us a shared vocabulary (GET/POST/PUT/DELETE, status codes, JSON) so a developer fluent in one RESTful API can productively consume any other within hours. This is exactly the leverage OneAPI exploits across ZIA, ZPA, ZDX, and Client Connector.

ASK A QUESTION
MY NOTES
💭THINK FIRST · WHAT IS AN API

Every day you use dozens of APIs without thinking about it — when your phone shows weather, when you pay with Apple Pay, when you share a post. Think about the structure of those interactions before we define them formally.

  • In the 'waiter' analogy, the waiter takes your order to the kitchen. But who decides the menu of what's available to order — and how does that map to an API?
  • What is the minimum information a server needs to know before it can respond to an API request? Think about it like receiving a letter — what must be on the envelope?
  • If a public API gets overloaded with millions of requests, what might the server return — and what should the client do with that response?
Q1 — MODEL ANSWER
The "menu" is the API specification — the documented set of endpoints, methods, and parameters the server is willing to accept. The waiter (API) can only relay what the kitchen (backend) has been told to support, which is why API documentation and OpenAPI/Swagger specs matter so much. If you ask for something not on the menu, you get a 404 or 400 — not a custom dish.
Q2 — MODEL ANSWER
At minimum the server needs: WHO (identity/auth, typically a Bearer token in the Authorization header), WHERE (the endpoint URL identifying the resource), WHAT (the HTTP method specifying the operation), and HOW (the request body and content type for POST/PUT). Without any of these the request fails — much like a letter missing a recipient address or stamp cannot be delivered. This is the four-part mental model worth committing to memory.
Q3 — MODEL ANSWER
An overloaded API typically returns a 429 Too Many Requests (or 503 Service Unavailable), often with a Retry-After header indicating when to try again. A well-behaved client should not retry immediately in a tight loop — it should back off exponentially, respect the Retry-After value, and queue work rather than hammer the server. Zscaler's per-method rate limits in OneAPI exist precisely so clients learn to design with these constraints in mind.
WHAT IS AN API

API Fundamentals

An API (Application Programming Interface) is a set of rules and protocols that allows different software applications to communicate with each other. It defines the methods and data formats applications use to request and exchange information.

Core definition: An API is the messenger between applications. It takes a request from one system, tells another system what to do, and returns the response — like a waiter taking your order to the kitchen and bringing back your food.

How APIs Work: Request-Response Cycle

Every API interaction follows the same pattern. A client sends a request to a server, the server processes it, and returns a response.

ComponentRoleExample
API ClientSends the API requestPostman, your script, Zscaler portal
EndpointThe URL that receives the requestapi.zsapi.net/zia/api/v1/users
HTTP MethodThe type of operationGET, POST, PUT, DELETE
Request HeadersMetadata about the requestAuthorization, Content-Type
Request BodyData sent with the request (JSON)User details for POST request
API ServerAuthenticates, validates, retrieves dataZscaler backend service
Response HeadersMetadata about the responseContent-Type, Rate-Limit info
Response BodyThe returned data (JSON)User list, policy details

The kitchen never talks to you directly — just like the database never talks to your browser directly.

ASK A QUESTION
MY NOTES
💭THINK FIRST · RESTFUL APIS

"Stateless" is one of the core REST principles — and one of the hardest to fully grasp. Think about what "state" means in a conversation, and what it would mean for an API to have zero memory between calls.

  • If a REST API is stateless and remembers nothing between requests, how does it know you're still authenticated on your second API call — given it has no memory of the first?
  • "Layered system" means the client doesn't know if it's talking to the actual server or a proxy in between. Why is this an advantage rather than a problem — what does the client gain from not knowing?
  • Caching GET responses improves performance — but what kind of data would be dangerous to cache, and why?
Q1 — MODEL ANSWER
"Stateless" means the server keeps no per-client memory between calls — so every request must carry its own proof of identity. That's exactly why the Bearer token (or API key) is included in the Authorization header of every single request, not just the first one. The token itself is the state; the client carries it around so the server doesn't have to remember anything about the session.
Q2 — MODEL ANSWER
The client benefits from layering because the architecture behind the endpoint can evolve — load balancers, caches, gateways like OneAPI, even backend re-platforming — without breaking client code. If the client knew it was talking to a specific server, every infrastructure change would require client updates. Hiding the intermediaries is what lets Zscaler add capacity, route through OneAPI, or swap implementations transparently.
Q3 — MODEL ANSWER
Anything personalized, security-sensitive, or rapidly changing is dangerous to cache: per-user policy results, real-time threat telemetry, authentication tokens, or session-specific data. If a shared cache serves stale or another user's data you get both a correctness and a security incident. The safe pattern is to cache only what is genuinely public and stable, and to use cache-control headers and short TTLs for everything else.
RESTFUL APIs

RESTful API Principles

A RESTful API (Representational State Transfer) uses standard HTTP methods to perform operations. All Zscaler APIs are RESTful, meaning they follow a consistent set of architectural constraints.

Stateless
Each request contains all information needed. No session state is stored on the server between requests.
Client-Server
Client and server are separate. Client handles UI/UX; server handles data and business logic.
Cacheable
Responses can be cached to improve performance. OneAPI uses caching for frequently requested data.
Uniform Interface
Consistent way to interact with resources. All Zscaler APIs follow the same patterns and conventions.
Layered System
Client doesn't need to know if it's talking to the actual server or an intermediary (like OneAPI Gateway).
i
WHY THIS MATTERS FOR ONEAPI
Because all Zscaler APIs follow RESTful principles through OneAPI, a developer only needs to learn one pattern. The same request structure, authentication approach, and response format works across ZIA, ZPA, ZDX, and Client Connector APIs.
Analogy
A stateless API is like a vending machine. Every time you interact with it, you insert money and select your item — it doesn't care that you used it yesterday. It gives you no loyalty discount. Each interaction is completely self-contained. That's why every API call includes the Bearer token: the vending machine needs your coins every single time.

ASK A QUESTION
MY NOTES
💭THINK FIRST · HTTP METHODS

Four HTTP methods — GET, POST, PUT, DELETE — but their rate limits are dramatically different. DELETE gets 4 requests per hour. GET gets 4,000. Before reading, think about why Zscaler would make DELETE 1,000x more restrictive than GET.

  • Why is it important that GET is "read-only" and cannot change data — what problem would occur if a read operation had side effects?
  • DELETE is limited to 4 requests per hour — that seems extremely conservative. Under what real scenario would someone need to make more than 4 DELETE calls per hour, and why might Zscaler want to slow that down?
  • PUT updates an entire resource — if you only want to change one field, what must your request body contain, and why might that be a problem?
Q1 — MODEL ANSWER
In REST, GET is defined as "safe" and "idempotent" — calling it any number of times should not change server state. This matters because intermediaries (browsers, proxies, OneAPI gateway, monitoring tools) freely retry, prefetch, and cache GETs. If GET had side effects, simply refreshing a page or a probe by a monitoring system could accidentally create, update, or delete data. Reserving mutation strictly for POST/PUT/DELETE keeps the system predictable.
Q2 — MODEL ANSWER
DELETE is permanent and almost impossible to undo at the API layer, so legitimate bulk-delete scenarios — like cleaning up stale user objects or removing policy rules — are rare and should be deliberate. A loop that DELETEs hundreds of items per minute is far more often a runaway script or a credential leak than a real workflow. The aggressive limit (4/hr) is a safety throttle: it forces engineers to batch through alternate workflows or open a case, and it caps the damage from an attacker who steals a token.
Q3 — MODEL ANSWER
PUT is a full replacement, so your body must contain the entire object including the unchanged fields — not just the field you care about. The problem is twofold: you must first GET the current object to avoid wiping fields you didn't touch, and you risk a race condition if someone else updates the resource between your GET and your PUT. This is why PATCH (partial update) exists in some APIs, and why "GET then PUT" patterns should use ETags/version numbers to detect conflicts.
HTTP METHODS

HTTP Methods & Request Structure

APIs use HTTP methods to indicate the desired action. Each method maps to a CRUD operation and has a specific rate-limit weight in Zscaler's OneAPI.

GET
Retrieve Data
Read-only. Retrieves resources without modifying them. Light weight — up to 4,000 req/hr in OneAPI. Example: list all users, get policy details.
POST
Create Resource
Creates a new resource on the server. Medium weight — 1 req/sec, 400 req/hr. Example: add a URL filtering rule, create a user.
PUT
Update Resource
Updates an existing resource in its entirety. Medium weight — same limits as POST. Example: update a policy rule configuration.
DELETE
Remove Resource
Deletes a resource from the server. Heavy weight — 1 req/min, 4 req/hr. Use carefully — irreversible action.

API Request Structure

Endpoint URL
The address of the API resource. For Zscaler OneAPI: https://api.zsapi.net/{product}/api/v1/{resource}
HTTP Headers
Metadata sent with the request. Key headers: Authorization: Bearer {token}, Content-Type: application/json
Request Body (JSON)
Data payload for POST/PUT requests. Must be valid JSON. Zscaler provides JSON templates to standardise configurations.
API request structure showing endpoint, method, headers, and JSON body components

PUT = replace a book with a revised edition (moderate restriction). DELETE = permanently remove a book from existence forever (very restricted — hard to undo, requires approval).

ASK A QUESTION
MY NOTES
💭THINK FIRST · WHY APIS MATTER

Imagine an enterprise with 100 branch sites, 50,000 users, and a security team of 10 people. A new threat emerges and every site needs a policy update. Without APIs — just manual portal clicks — think through the math of what "fast response" actually means.

  • If updating a URL filtering policy manually takes 5 minutes and you have 100 sites, how long does the full rollout take — and what's the risk of inconsistency between the sites updated first versus last?
  • "Reduced human error" is one of the automation benefits. But automation scripts can also have bugs. What practices would make an automation script more reliable than manual configuration?
  • "Faster threat response" is only valuable if the automation is triggered quickly. What needs to be in place for a new threat indicator to automatically block traffic within minutes of discovery?
Q1 — MODEL ANSWER
At 5 minutes per site times 100 sites you are looking at roughly 8.3 hours of pure click time — and in practice much longer once you account for context switching, breaks, and verification. The bigger risk is the window of inconsistency: for most of that day, the first sites updated are protected while the last sites still carry the old policy, leaving an exploitable gap. APIs collapse the window from hours to seconds and make the change atomic across the fleet.
Q2 — MODEL ANSWER
Reliability comes from engineering discipline applied to scripts: version control, peer review, automated tests, dry-run/preview modes, idempotency, structured logging, and staged rollouts (canary → batch → full). A script that has been reviewed, tested in a sandbox tenant, and rolled out to 1 site before 1,000 is far safer than a human clicking through the UI under deadline pressure. Manual work fails silently; automation can be designed to fail loudly and rollback automatically.
Q3 — MODEL ANSWER
You need a full detection-to-response pipeline: a trusted threat-intelligence source emitting IOCs, a SIEM or SOAR system that ingests and validates them, integration credentials (ZIdentity OAuth tokens) with appropriate scopes, a documented playbook describing exactly which Zscaler API calls to make, and guardrails like allowlists and approval thresholds to prevent false positives from blocking legitimate traffic. Speed without these controls turns an automation win into an outage.
WHY APIS MATTER

Why APIs Power Zero Trust Automation

APIs are the foundation of the Zscaler Zero Trust Automation strategy. Without APIs, every configuration change would require manual portal interaction — slow, error-prone, and impossible to scale.

Integration
Connect Zscaler with SIEM, ITSM, and third-party tools. ServiceNow + ZDX is a key example.
Automation
Automate repetitive tasks — policy deployment, user provisioning, URL filtering updates.
Monitoring
Pull real-time health, performance, and security telemetry programmatically.
Authentication
ZIdentity OAuth tokens allow secure, controlled programmatic access to all Zscaler services.
Developer Engagement
Partners and customers can build on top of Zscaler's platform using documented, stable APIs.
