COURSE SUMMARY

In Summary

In this course, you started with an overview of APIs, covering their key features and how they work. You then transitioned to an in-depth look at OneAPI, focusing on ZIdentity for Zscaler Zero Trust Automation. You explored the fundamental concepts of ZIdentity and how it simplifies access to Zscaler services and facilitates Zscaler Zero Trust Automation. You were then presented with a detailed, step-by-step guide for accessing OneAPI, ensuring that participants have all the tools to utilise it effectively.

Furthermore, you explored the fundamental concepts of API architecture, which set the stage for understanding communication between different Zscaler services. You then discussed the workflow of traditional Zscaler Automation Framework and identified the common use cases of Zscaler Zero Trust Automation. Lastly, you covered common API error codes and best practices to enhance efficiency, maintain security, and streamline operations.

Section 1 — API Overview
Provided the key aspects of API, how it works, and how it is different from RESTful APIs. Covered endpoints, HTTP methods (GET/POST/PUT/DELETE), request/response cycle, status codes, and why APIs are essential for automation.
Section 2 — Zero Trust Automation Platform
Provided an understanding of ZIA, ZPA, ZDX, Client Connector, and Branch/Cloud Connector APIs. Covered the Zero Trust Automation Framework using OneAPI and ZIdentity — including fundamental aspects of ZIdentity, authentication, provisioning, API clients, API resources, and access tokens. Demonstrated configuring OneAPI using Postman REST API client.
Section 3 — Architecture, Framework and Use Cases
Explored the architecture of Zscaler API, set the stage for understanding communication between different Zscaler services. Discussed the workflow of traditional Zscaler Automation Framework. Discussed ZIA URL blocking, ZPA App Connector group creation, and ZCC device management use cases demonstrating real-world automation.
Section 4 — Troubleshooting and Best Practices
Covered common API response codes (200–503). Discussed rate limiting with the two-tier system (burst and volume limits). Covered best practices: secure API usage, maintaining consistency, enhancing scalability, and deployment methodologies. Provided systematic error debugging approach.

API FUNDAMENTALS
ConceptDefinition / Key Points
APISet of rules/protocols for software to communicate. Defines methods and data formats for requesting/exchanging information.
RESTful APIUses standard HTTP methods. Stateless, client-server, cacheable, uniform interface, layered system.
GETRead/retrieve data. Light weight: 2 req/sec, 4,000 req/hr.
POSTCreate a new resource. Medium weight: 1 req/sec, 400 req/hr.
PUTUpdate an existing resource. Medium weight: same as POST.
DELETERemove a resource. Heavy weight: 1 req/min, 4 req/hr.
Status 200/201Success — data returned / resource created.
Status 401/403Auth failed (401) / forbidden — check token, role, SKU (403).
Status 429Rate limit exceeded. Check Retry-After header. Add sleep() to scripts.

APIPurposeBase Path
ZIA APICloud Service, Sandbox Submission, 3rd-Party App Governance/zia/api/v1

ZIDENTITY
FeatureDetail
PurposeUnified identity service — centralises auth, entitlements, and provisioning for all Zscaler services.
Auth ProtocolOIDC (OpenID Connect) for SSO. OAuth 2.0 for programmatic access.
ProvisioningSCIM automates user provisioning/deprovisioning across all ZIA and ZPA tenants simultaneously.
MFARequired by default. Supports SMS OTP, email OTP, Google Authenticator (TOTP), FIDO.
Token EndpointPOST to https://<Vanity Domain>.zslogin.net/oauth2/v1/token
ONEAPI
AspectDetail
Hostapi.zsapi.net — single endpoint for all Zscaler APIs
Globally DistributedAPI gateways in multiple regions. Low latency via nearest gateway.
Geographic AwareGeoProximity routing to nearest regional gateway.
Cloud AgnosticSingle endpoint for all product clouds — no multiple URL management.
Authenticate OnceZIdentity OAuth token provides access to multiple services.
Steps to Access1. Locate base URL → 2. Authenticate and retrieve token → 3. Make API call with Bearer token
ComponentsAPI Client (registered in ZIdentity), API Resources (scoped endpoints), Access Tokens (time-sensitive JWTs)
BEST PRACTICES
CategoryPractice
Secure UsageRotate tokens. Separate tokens per endpoint. Validate/assign specific resources. Never share Secret ID/JWT/Client ID. Always HTTPS.
ConsistencyUse Zscaler JSON templates. Implement version control for all API scripts.
ScalabilityBatch requests for bulk ops. Monitor usage via Postman to avoid rate limits.
DeploymentAlways test in staging before production. Use environment variables for sensitive info.
Error DebuggingCheck JSON formatting → Verify headers/body/params/method → Check auth state → Handle 429 with Retry-After → Retry 409 after delay

MEMORY TIP — THE 3 STEPS TO ONEAPI
1. Locate — Find your base URL (api.zsapi.net/{product}/api/v1). 2. Authenticate — POST to ZIdentity token endpoint with your Client ID and Secret to get a Bearer token. 3. Call — Include Authorization: Bearer {token} in every API request header. That's it — OneAPI handles the rest.

CRITICAL FACTS TO KNOW
OneAPI Host
api.zsapi.net — universal host for all Zscaler API products via OneAPI.
Token Endpoint Format
https://<VanityDomain>.zslogin.net/oauth2/v1/token — POST with Client ID + Secret to get Bearer token.
ZIA Rate Limits
GET: 2/sec, 4000/hr. POST/PUT: 1/sec, 400/hr. DELETE: 1/min, 4/hr.
403 Causes (4)
API key disabled. No role permissions. Missing SKU. Portal in maintenance mode during write ops.
3 Access Steps
1. Locate base URL. 2. Authenticate + get token from ZIdentity. 3. Make API call with Bearer token.
