ZCC — Exporting Logs

Collecting log bundles, packet captures, and connector evidence from the client side
"Every ZCC failure has a fingerprint on disk. Knowing exactly where to copy it from — and what to ship with it — is what turns a vague complaint into a solved ticket."
💭 THINK FIRST · EXPORTING ZCC LOGS

The Zscaler Client Connector log export is the single most useful artifact you can hand to Zscaler support. It captures authentication state, posture results, tunnel decisions, and per-app session attempts — all from the user's actual device, at the moment of failure. But it only works as evidence if it's collected at the right time.

  • If a user encounters a ZCC issue and immediately hits "Export Logs" from the system tray, what does the export contain — and what's missing if they wait an hour?
  • Why does the recommended workflow ask you to start a packet capture before reproducing the problem, even though the ZCC log will already show the problem on its own?
Q1 — MODEL ANSWER
An immediate export captures the live tunnel state, the most recent authentication and posture transactions, and the rolling log buffer with the failure inside it. Wait an hour and that buffer rolls over: a busy ZCC easily writes through its log limit, the relevant lines age out, and you're left with logs that show only "things are working now". Worse, if the user logs out and back in, the failure event may be permanently lost. Speed of capture matters more than completeness of analysis at the moment.
Q2 — MODEL ANSWER
The ZCC log shows ZCC's view of events — but if the failure is on the wire (a SYN that gets no SYN-ACK, an RST from an upstream firewall, a TLS handshake that's reset), ZCC just sees "the connection failed". A parallel packet capture shows the actual bytes: was there a SYN-ACK? Was there a TCP reset, and with which TTL? That's why support recommends running both: ZCC tells you what was attempted; the PCAP tells you what was returned.
SECTION 1

Exporting Zscaler Client Connector Logs

The ZCC log export is the canonical "bundle for support" from the user's device. The workflow is: collect logs first by clearing and starting a packet capture, then reproduce the problem, then export the logs for support.

The full ZCC and App Connector log-export procedure as documented in the course: clear logs · start the packet capture · reproduce the problem · export ZCC log via the system tray icon · for ZPA App Connector access, SSH in as admin and run journalctl with reproduce-and-bundle.

Procedure

STEP 1
Clear logs
From a clean baseline. Everything from here is causally tied to the reproduction.
STEP 2
Start packet capture
Wireshark on Windows/macOS or tcpdump on Linux. Save to disk with a recognisable filename.
STEP 3
Reproduce
Do the exact action the user reported failing. Note the wall-clock time.
STEP 4
Export logs
Right-click the ZCC tray icon → Export Logs. Stop the packet capture.
STEP 5
Bundle and submit
ZCC log zip + PCAP + screenshot + timestamp + user identifier. Attach to ticket.
i
WHY THIS ORDER
Capturing the packet trace before the export means the PCAP and the ZCC log share a timestamp range. Support correlates "this failed ZCC session" against "this TLS handshake that got an RST" in seconds — instead of trying to align two artifacts that overlap by accident.
🧠
MENTAL NOTE
ZCC log export = right-click tray icon → Export Logs. Pre-export order: clear → PCAP-on → reproduce → export → PCAP-off → bundle. Submit ZCC log + PCAP + screenshot + timestamp + user ID. Speed matters; the log buffer rolls over.

💭 THINK FIRST · APP CONNECTOR LOGS

App Connector log collection happens over SSH on the connector itself, not from the ZPA admin portal. That's a deliberate design choice: when a connector is unhealthy, the admin portal sees only "Disconnected" — but the connector still has detailed local logs about why it disconnected.

  • If an App Connector shows "Disconnected" in the ZPA admin portal but ping to the connector's IP from your laptop succeeds, what does that combination tell you?
  • Why is "free up space on /opt/zscaler" a frequent first remediation step for App Connector failures?
Q1 — MODEL ANSWER
It tells you the connector host is up and reachable, but its outbound tunnel to the upstream ZPA Public Service Edge has broken. The connector dials out — it doesn't accept inbound from ZPA — so it can be perfectly responsive on its own subnet while completely unable to register. The next step is to verify the connector's outbound path: DNS resolves to broker.prod.zpath.net, and the firewall match goes to the ZPA range config.zscaler.com/private.zscaler.com/zpa.
Q2 — MODEL ANSWER
Upgrade failures often leave behind copies of /opt/zscaler/var/version and /opt/zscaler/var/updater.version that fill the connector's small partition. Once the disk is full, the connector can't write logs, can't complete upgrades, and may fail to start the tunnel. Clearing extra version directories and removing /opt/zscaler/var/image.bin reclaims space and lets the connector recover without a full reinstall.
SECTION 2

App Connector — Common Issues & Logs

App Connectors are Linux hosts deployed close to the private application. Most issues fall into two buckets: Upgrade Failure (disk full, stale binaries) and Disconnected (outbound path to the ZPA Public Service Edge has broken).

The course's reference table for the two most common App Connector failures and their remediation steps — Upgrade Failure (collect version outputs, free disk space, delete stale image binaries) and Disconnected (test upstream Service Edge connectivity, verify firewall match against ZPA IP ranges).

App Connector Log Collection (over SSH)

  1. SSH into the App Connector as admin.
  2. Initiate packet capture: tcpdump -i <iface> -w /var/log/messages/<relevant_file_name> <filter>
  3. Press Ctrl+C on the App Connector to stop the packet capture once the issue has been reproduced.
  4. Enter the journal command: cd /var/tmp then run journalctl -p info --since <time> > journal_logs.txt
  5. Run the command cat /var/log/messages/* and copy the messages from the last 2 minutes of the duration when the test was made — paste into your local machine to share with support.
  6. Upload ZCC exported logs, connector logs (PCAPs) and journal logs to https://tools.zscaler.com/upload (password P5Fg9SNCVxpLqUFJ) and once done, share the link.
!
UPGRADE FAILURE — RECOVERY
Collect sudo cat /opt/zscaler/var/version, sudo ls /etc/yum/repos.d/zscaler.repo, and sudo /opt/zscaler/var/updater.version. If the connector can't download a new image because disk is full, delete extra version directories and /opt/zscaler/var/image.bin to free space.
!
DISCONNECTED — RECOVERY
Test connectivity to the upstream ZPA Public Service Edge by reaching broker.prod.zpath.net from the connector. Verify the connector's source IP is in the firewall match against the ZPA range at config.zscaler.com/private.zscaler.com/zpa.
🧠
MENTAL NOTE
Two main App Connector failures: Upgrade Failure (disk-full and stale binaries — free /opt/zscaler, delete image.bin) and Disconnected (broker.prod.zpath.net unreachable — verify outbound firewall). App Connector logs come from journalctl + tcpdump over SSH, NOT from the ZPA admin portal. Upload bundle: tools.zscaler.com/upload.

💭 THINK FIRST · PERFORMANCE TRIAGE

"Performance is bad" is rarely a single problem. It's a vague symptom that resolves into one of several specific paths once you ask better questions: client to ZPA Service Edge, App Connector to ZPA Service Edge, or App Connector to the application server. Each path has its own diagnostic recipe.

  • If a user reports slow access to ONE private app but other private apps work fine, which path is the most likely suspect — and which path can you essentially rule out?
Q1 — MODEL ANSWER
Other apps working fine means the client-to-Service-Edge path and the Service-Edge-to-Connector tunnel are healthy — those are shared by all private apps and would degrade together. The suspect path is App Connector to the specific application server: a ping test and ICMP MTR from the App Connector toward that server's IP, plus a packet capture on the App Connector while a user accesses the app, will isolate whether the slowness is on the connector-to-server leg or inside the application itself.
SECTION 3

Performance Triage: 3 Paths, 3 Recipes

Performance issues in a ZPA path could potentially reside on one of three legs. Each leg has its own troubleshooting recipe to isolate where the issue is originating from.

PATH 1
Client → ZPA Service Edge
Ping test · ICMP MTR · throughput test from a server near the Service Edge's geographic location · packet capture from the ZCC while accessing the application.
PATH 2
App Connector → ZPA Service Edge
Ping test · ICMP MTR from the App Connector. Confirms the connector's outbound tunnel leg is healthy.
PATH 3
App Connector → App Server
Ping test · ICMP MTR · packet capture from the App Connector while a user accesses the application. Isolates server-side issues.

Client-Side Performance Sanity Check

Before blaming Zscaler, run Task Manager on Windows clients. The Performance tab gives a quick read on whether the client itself is bottlenecking the experience — CPU, memory, disk, and Wi-Fi throughput in one view.

Windows Task Manager Performance tab — quick sanity check on whether the client itself is bottlenecking. Download a 100 MB test file (e.g. azurespeed.com) and watch the throughput graph and resource utilisation with and without Zscaler. Big drops with Zscaler enabled point to inspection load; equal drops in both cases point to the local network or device.

The Performance Triage Questionnaire

  • Is this a new issue or an existing one?
  • Are other users affected? What's the common pattern among them?
  • What kind of internet link is the user connected to — ethernet, Wi-Fi?
  • Is the user accessing from a single location, or moving between locations (home, office, hotel)?
  • Is the issue regardless of location — does it follow a single user across networks?
  • Is the issue specific to a single app, single connector group, or present everywhere on ZPA?
  • Is the issue present when ZPA is not connected (i.e. accessing internet directly)? If yes, it's not ZPA.
  • Is there a different method of accessing the same application that shows similar symptoms?
  • Is there more than one internet link at the site? Does the issue change on the alternate link?
i
NARROWING THE WINDOW
If the issue appears only when ZPA is enabled, the cause sits inside the ZPA path — work the three legs above. If the issue happens also when ZPA is disconnected, ZPA is innocent — the problem is the device, local network, or upstream ISP.
🧠
MENTAL NOTE
Performance issues live on one of three legs: (1) Client → ZPA Service Edge, (2) App Connector → ZPA Service Edge, (3) App Connector → App Server. Each leg's recipe = ping + ICMP MTR + PCAP. First triage question: does the problem persist with ZPA disconnected? If yes, ZPA is not the cause.
Analogy
Triaging a slow ZPA path is like diagnosing a slow water tap with three pipe segments in series: the street main, the house feed, and the kitchen pipe. You don't replace the whole plumbing — you test each segment separately. If the street pressure is fine and the house feed is fine, then the kitchen pipe is the suspect. ZPA is the same: ping each leg, find the slow one, fix only that leg.

💭 THINK FIRST · WIRESHARK RECIPES

A packet capture without a filter is just noise. The course Appendices distill the seven most common Wireshark/packet-capture investigations into specific filter strings — each one designed to answer a specific question fast.

  • If you see TCP retransmissions and RSTs coming back with a TTL that differs from the rest of the trace, what does that imply about who is sending the reset?
  • Why does the SSL-issues recipe filter for both tls.handshake.type eq 1 and frame contains xyz.com together, rather than just one of them?
Q1 — MODEL ANSWER
A different TTL on RST packets indicates the resets are originating from a different number of hops away than the legitimate traffic — meaning a middlebox (firewall, IPS, transparent proxy) is forging RSTs to tear down the session, while the real endpoint's packets travel a different distance. This is the classic signature of an inline policy device blocking the connection rather than the server refusing it. Same source IP, different TTL = forged reset.
Q2 — MODEL ANSWER
Filtering only on TLS handshake type 1 gives every Client Hello in the trace — far too many to read. Filtering only on the hostname misses the Client Hello itself, which is unencrypted and contains the SNI you actually want to inspect. The compound filter narrows the result to the handshakes for the impacted hostname, so you immediately see the start of the failed TLS conversation and the response (or RST) that followed it.
SECTION 4

Wireshark Recipes (Course Appendices 1–7)

Seven targeted packet-capture investigations, each tied to a question you commonly need to answer when handing evidence to support.

Appendix 1 — Connections to a Specific Website/IP

  • Filter HTTP requests: http.request.method
  • Filter HTTPS Client Hellos: tls.handshake.type eq 1
  • Specific IP: ip.addr eq x.x.x.x
  • Apply host in the HTTP request column and Servername (SNI) in the Client Hello column. Right-click the request and Follow the TCP Stream.

Appendix 2 — Firewall or Local Network Blocks

  • Look for SYN packets to a specific IP with no SYN-ACK coming back.
  • Check for reset packets with a TTL different from the other packets received from Zscaler — a sign of a forged RST from an inline middlebox.
A forged-RST signature in Wireshark: the TLS handshake establishes with TTL 128, but the reset packets arrive with TTL 118 — confirming the reset came from an inline device (firewall/IPS), not the actual server.

Appendix 3 — DNS Issues

  • Filter for DNS protocol: dns
  • Optional: dns contains "xyz.com" to scope to one domain.
  • Follow the UDP Stream and check the DNS response for the resolved IP.
Following a DNS UDP stream in Wireshark: filter for the DNS protocol (optionally narrowed by hostname), right-click a query, choose Follow → UDP Stream. The response reveals the resolved IP that the client actually used — a quick way to confirm DNS poisoning or split-DNS misconfig.

Appendix 4 — SSL Issues

  • Filter for the impacted site: tls.handshake.type eq 1 and frame contains "xyz.com"
  • Follow the stream and look for SSL errors — certificate, Encrypted Alerts. SSL failures usually close with an RST or FIN-ACK immediately after the handshake messages.

Appendix 5 — File Download or Webpage Load Time

  • Apply Appendix 1 filter for the specific page or test file.
  • Right-click the request and Follow the TCP stream.
  • Statistics → Capture File Properties → Statistics → Displayed — observe the time taken and average bits/s, with and without Zscaler.

Appendix 6 — Client Performance via Task Manager

  • Open Task Manager → Performance tab.
  • Download a 100 MB test file from azurespeed.com or any test site.
  • Monitor the throughput graph and download speed with and without Zscaler.
  • Track CPU, Memory, Disk utilisation in parallel.

Appendix 7 — Page Loading Time (Browser DevTools)

  • In Chrome: More tools → Developer tools (or Ctrl+Shift+I).
  • Start the trace and open the impacted site.
  • Click the relevant request and select the Timings tab.
  • Note DNS lookup time, SSL handshake, and response times.
🧠
MENTAL NOTE
7 PCAP recipes: (1) Specific IP — tls.handshake.type eq 1 + ip.addr eq x.x.x.x; (2) Firewall blocks — SYN without SYN-ACK; check RST TTL mismatch; (3) DNS — dns filter, Follow UDP Stream; (4) SSL — handshake + frame contains; (5) Download time — Statistics → Capture File Properties; (6) Client throughput — Task Manager + 100MB azurespeed; (7) Page load — DevTools Timings.
