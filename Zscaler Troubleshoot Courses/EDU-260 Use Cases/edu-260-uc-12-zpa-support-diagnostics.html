ZPA Support & Diagnostics

Clearing logs, reproducing problems, and packaging clean evidence for support
"The difference between a 30-minute ticket and a 30-day ticket is whether you sent clean logs from a reproduced problem — or muddy logs from a long-running working session."
💭 THINK FIRST · ZPA DIAGNOSTIC LOGS

ZPA diagnostic logs sit between two extremes: too little data and you can't see what failed; too much data and you drown in noise from sessions that worked fine. The art of using them well is in controlling the window — when did logging start, when was the problem reproduced, when did logging stop.

  • Why is "clear the logs, then reproduce the problem" almost always the first thing Zscaler support asks you to do — what changes about the data quality?
  • If a user reports "ZPA was slow this morning at 9 am" and it's now 3 pm, what's the realistic value of the logs you collect right now versus the value of asking the user to reproduce?
Q1 — MODEL ANSWER
A long-running ZPA session generates thousands of routine log lines — DNS lookups, tunnel keepalives, successful posture checks. The actual failure is a needle in that haystack, and worse, the cause may have rolled out of the log buffer. Clearing logs creates a known-good baseline: every entry after the clear is causally linked to the action you took to reproduce. Support engineers can correlate timestamps across ZIA, ZPA, ZCC, and packet captures with confidence.
Q2 — MODEL ANSWER
Logs collected six hours after the fact are mostly useless. ZCC and App Connector logs roll over, the local DNS cache and MUP cache have been refreshed, and the network conditions at 9 am may be entirely gone. Asking the user to reproduce — even if the problem is intermittent — gives you a chance to capture a fresh, time-bounded, multi-source bundle. If the issue won't reproduce, that itself is diagnostic information: it points to a transient cause like an upstream outage rather than a persistent misconfiguration.
SECTION 1

ZPA Diagnostic Logs — What They Capture

Zscaler Private Access exposes several layers of diagnostic data — from session status codes returned by Service Edges, to App Connector health, to user-facing logs in the Zscaler Client Connector. Each layer answers a different question, and you need the right one for the symptom you're seeing.

The Three Diagnostic Surfaces

ZPA SESSION STATUS
help.zscaler.com/zpa/about-zpa-session-status-codes
Numeric codes returned per session attempt — they pinpoint whether the failure was at policy evaluation, App Connector routing, posture check, or server connectivity. The reference list of codes is the first lookup when a session won't establish.
CONNECTOR DIAGNOSTICS
help.zscaler.com/zpa/about-connector-diagnostics
Health, latency, and reachability data exposed by each App Connector. Tells you whether the connector can reach its target server and whether the upstream Service Edge tunnel is healthy.
APP CONNECTOR TROUBLESHOOT
help.zscaler.com/zpa/troubleshooting-app-connectors
The reference for connector-side failures — upgrade failures, disk full, "Disconnected" status. Use this when ZPA users can authenticate but no application traffic flows.

Reading a ZPA Failure: Locate, Isolate, Diagnose

Every ZPA troubleshooting case follows the same shape: figure out which segment of the path is broken, then collect the right log surface for that segment.

LOCATE
Where in the path?
Client → Service Edge → App Connector → Server. Confirm authentication succeeded (ZCC status), then check whether the session ever reached the App Connector (session status code).
ISOLATE
One user, one app, one site
Narrow the scope before pulling logs. Is it everyone or one user? Is it one app or all private apps? Does it happen from the office and from home, or just one location?
DIAGNOSE
Pull the matching log
Authentication problem → ZCC + IdP logs. Routing problem → ZPA session status codes. Server unreachable → App Connector diagnostics on the connector serving that segment.
EVIDENCE
Clean bundle for support
Clear logs · reproduce · export ZCC logs and packet capture · attach to ticket with exact timestamp of the reproduced failure.
i
RULE OF THUMB
Three reference URLs to bookmark for any ZPA case: about-zpa-session-status-codes, about-connector-diagnostics, troubleshooting-app-connectors. Most ZPA tickets are resolved by mapping a numeric session status code to one of these pages.
🧠
MENTAL NOTE
ZPA diagnostics live in 3 places: session status codes (per-session, numeric, on help.zscaler.com/zpa/about-zpa-session-status-codes), connector diagnostics (per-connector health), and App Connector troubleshooting reference. Workflow = Locate → Isolate → Diagnose → Evidence. Always clear logs and reproduce; never submit stale logs.

💭 THINK FIRST · THE SUPPORT WORKFLOW

Zscaler support engineers work the same case dozens of times a day. The single biggest accelerator of a ticket is not the eloquence of your description — it's whether the attached evidence shows the problem happening, with timestamps, in a clean log window.

  • Why does restarting the network adapter (not just clearing logs) matter before reproducing — what state does it actually reset?
  • If you can only attach one piece of evidence to a ZPA ticket, should it be the ZCC log bundle or the packet capture, and why?
Q1 — MODEL ANSWER
Restarting the network adapter clears the local DNS cache and the MUP (Microsoft User Profile / micro-tunnel) cache, plus any cached DNS responses that ZPA is relying on for app discovery. If a stale DNS entry is the actual cause of the failure, clearing only the ZCC logs would still leave the bad cached entry in place — you'd reproduce the symptom but never see the corrected path. Restarting the adapter forces a clean DNS state, so the reproduced session reflects current configuration.
Q2 — MODEL ANSWER
For most ZPA failures the ZCC log bundle is more valuable than a raw packet capture, because the ZCC bundle already contains the session-level context — which app was attempted, what status code came back, which connector served it. A packet capture is bytes on the wire without that interpretation, and on ZPA most traffic is double-encapsulated and encrypted, so the capture alone is hard to parse. Send both when you can, but if forced to choose, send the ZCC bundle.
SECTION 2

The Clean-Bundle Support Workflow

When you're handing a real problem to Zscaler support, the goal is to deliver a clean, time-bounded, multi-source evidence bundle. The recommended steps are deliberately ordered so that each piece of evidence is consistent with the others.

STEP 1
Clear logs
Start from a known-good state. Everything captured from here is causally linked to the reproduction.
STEP 2
Restart network adapter
Flushes the local DNS cache and MUP cache, plus other system-level network state that can mask the real problem.
STEP 3
Start packet capture
Wireshark or tcpdump running before you reproduce. Captures the on-wire view that complements the ZCC log.
STEP 4
Reproduce the problem
Do the exact action the user reported failing. Note the wall-clock time — it becomes the anchor for cross-source correlation.
STEP 5
Stop and export
Stop the packet capture, export ZCC logs, export ZIA/ZPA logs from the admin portal for the reproduction window.
STEP 6
Attach to ticket
Bundle the PCAP, ZCC logs, screenshots, exact timestamp, user identifier, and target app name. Include the session status code if you have one.
!
COMMON MISTAKE
Submitting logs from "any time today" without a reproduction window. Support has to scan thousands of lines to find your problem. A 5-minute clean window beats a 5-hour noisy one — every time.

What Support Reviews From the Bundle

  • ZCC log — authentication state, posture results, session attempts and status codes.
  • Packet capture — DNS resolutions, TCP/TLS handshakes, retransmissions, RSTs from upstream.
  • ZPA portal log export — which Service Edge and Connector served the session, policy evaluation result.
  • ZIA portal log export — for hybrid users whose internet traffic also flows via ZIA; correlates DNS and threat-engine events.
🧠
MENTAL NOTE
Clean-bundle order: (1) clear logs · (2) restart network adapter · (3) start PCAP · (4) reproduce · (5) stop and export · (6) attach with timestamp. Always include user ID, app name, session status code, and exact reproduction time. Adapter restart matters because DNS cache and MUP cache survive a simple log-clear.
Analogy
A clean-bundle support ticket is like sending a crime-scene photo with a timestamp and a forensic ruler in the frame, instead of mailing a box of unsorted photos taken over a week. The detective doesn't have to guess what happened or when; they can see the moment, in context, and start work immediately.

💭 THINK FIRST · END-TO-END COVERAGE

By the time you finish a ZPA support workflow, you've crossed every diagnostic boundary in the Zscaler stack — end-user device, local network, internet, Zero Trust Exchange, application segmentation. Each boundary has its own logs. The skill is knowing which boundary the failure belongs to.

  • If users complain that "ZPA is broken" but ZIA works fine on the same device, what does that already tell you about which segment to investigate first?
Q1 — MODEL ANSWER
ZIA working while ZPA is broken on the same device rules out a whole class of problems: the device is online, ZCC is authenticated, the local network is reaching the Zero Trust Exchange, and identity is healthy enough for at least one service. That focuses you on ZPA-specific layers: ZPA policy and posture, Service Edge to App Connector tunnels, App Connector health, and the upstream private application itself. You can skip checking the user's home network and IdP and head straight for ZPA session status codes and connector diagnostics.
SECTION 3

End-to-End Diagnostic Coverage

The full Zscaler troubleshooting surface, from initial install to application connectivity, spans five layers. ZPA Support & Diagnostics is the fourth — but a good engineer reasons across all five.

LAYER 1
End-user device
ZCC installation, posture, identity. Symptoms: ZCC won't authenticate or stays disconnected.
LAYER 2
Local network
Home or branch network, captive portals, local DNS. Symptoms: ZCC connects intermittently or only on some networks.
LAYER 3
Internet path
ISP, transit, geographic routing to nearest Service Edge. Symptoms: latency spikes, region-specific failures.
LAYER 4
Zero Trust Exchange
ZIA and ZPA cloud services, policy evaluation, threat engines. Symptoms: blocked transactions, policy denials, session status codes.
LAYER 5
App segmentation
App Connectors, application segments, target server health. Symptoms: ZPA authenticates but app traffic times out.

What You Should Now Be Able To Do

  • Diagnose problems on the end-user device, local network, internet path, Zero Trust Exchange, and application segmentation — and know which logs belong to which layer.
  • Read ZIA logs and ZPA logs, and inspect Zscaler Client Connector logs from the device itself.
  • Run and read packet captures to trace authentication and connectivity flows from initial install through application access.
  • Package a clean evidence bundle for Zscaler support that they can act on inside a single review.
🧠
MENTAL NOTE
Five diagnostic layers: device → local network → internet → Zero Trust Exchange → app segmentation. ZPA Support & Diagnostics primarily covers layers 4 and 5. Always anchor your reasoning to a layer: don't pull connector logs for a problem that lives at layer 1, and don't dig into ZCC posture for an App Connector that's full disk.
