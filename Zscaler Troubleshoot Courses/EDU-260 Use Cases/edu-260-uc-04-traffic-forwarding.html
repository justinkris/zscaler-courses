EDU-260 · LESSON 4

Traffic Forwarding

Six common ZIA/ZPA/ZCC traffic forwarding failures — and the diagnostic path through each
"Auth lets the user in. Traffic forwarding gets the packet to where it's supposed to go — and that's where most outages live."
💭 THINK FIRST · WHICH FORWARDING METHOD

A Zscaler-bound packet can take one of several paths to the cloud: Tunnel 2.0 (DTLS/TLS), Tunnel 1.0, Tunnel-with-local-proxy, Enforce Proxy via PAC, or — if ZCC isn't installed — direct PAC/GRE/IPSec. The choice changes how you isolate every failure that follows.

  • Why does ZIA Tunnel 2.0 require UDP 443 (DTLS) to be allowed on the local network, and what's the safe fallback?
  • If a user reports "everything is broken", what's the single question that disambiguates ZCC vs PAC vs GRE/IPSec fastest?
Q1 — MODEL ANSWER
Tunnel 2.0 prefers DTLS for performance — it avoids TCP-over-TCP head-of-line blocking. Many corporate firewalls block UDP 443 by default. If DTLS can't establish, ZCC must fall back to TLS Tunnel 2.0; if that also fails, traffic forwarding breaks. The fix is either allowing UDP 443 outbound or pinning the app profile to TLS only.
Q2 — MODEL ANSWER
"What does the ZCC status panel say?" If ZCC reports Connected, it's a ZCC-mode user — work down the Tunnel/Enforce-Proxy branch. If ZCC isn't installed, it's PAC/GRE/IPSec — work down the PAC-URL/return-statement branch. One question collapses half the flowchart.
SECTION 1

Traffic Forwarding Flow

Once the requestor is authenticated, ZIA or ZPA need a way to receive the traffic. Zscaler recommends one of four mechanisms: tunnelling, PAC files, Zscaler Cloud Connector, or the Zscaler Client Connector. Each Service Edge is the point of delivery; ZCC is the primary mechanism for road-warrior and managed-device traffic.

The master Traffic Forwarding flowchart. Branches on ZIA vs ZPA; then ZCC vs BA (browser-only PAC); then tunnel mode (Tunnel 2.0 DTLS/TLS, Tunnel 1.0, Tunnel-with-local-proxy, Enforce Proxy).

Tunnel 2.0: verify UDP 443 (DTLS) allowed; check return statement & reachability to Service Edge.
Tunnel with local proxy: verify ${ZAPP_LOCAL_PROXY} macro in PAC return statement.
ZPA: check reachability to any.broker.prod.zpath.net.
Enforce Proxy / PAC: check PAC URL + contents; verify the proxy line.

💭 THINK FIRST · UC1 SITE UNREACHABLE

"Website unreachable through Zscaler" splits into two very different causes once you check the error: DNS (gateway timeout, NXDOMAIN) versus web server blacklisting (a 504 from a single DC). The same symptom; the fix depends entirely on which.

  • Why does a 504 from only one Zscaler DC suggest blacklisting rather than a Zscaler outage?
Q1 — MODEL ANSWER
If Zscaler infrastructure were broken globally, every DC would fail — not one. A single DC failing while the site is reachable via direct internet access and via secondary Zscaler DCs strongly suggests the destination is rejecting traffic originating from that specific Zscaler IP range. Third-party paid services sometimes blacklist cloud egress IPs to deter scraping. The fix is to route around the blacklisted node via PAC or contact the third party directly.
USE CASE 1 · ZIA TRAFFIC FORWARDING

ZIA Traffic Forwarding — Website Fails to Load

01 · PROBLEM
Connected to Zscaler but the site is unreachable
User is connected to the Zscaler service but the website doesn't load. Could be a gateway timeout or a DNS error.
02 · LOCATE
Inspect the actual error
Gateway timeout points to forwarding/path; DNS error points to resolution. Localize accordingly.
03 · ISOLATE
Four narrowing steps
1. Review provisioned domains (Admin Portal → Administration → Company Profile) and at the IdP verify UPN/Domain suffix.
2. If domain not provisioned, raise a provisioning ticket.
3. Validate other users can reach the website — narrows it to a specific Service Edge / DC.
4. Try a secondary Zscaler DC by editing the app-profile PAC file in ZCC.
04 · DIAGNOSE
Likely web server blacklisting
If DNS appears correct, the site is reachable via direct internet, but only a few Public Service Edges or a specific DC fail → likely web-server blacklisting of Zscaler IPs. Contact the third party. Workaround: route traffic away from the impacted Zscaler node or DC via the PAC file.
When a 504 appears for only specific DCs while other DCs work, the destination — not Zscaler — is rejecting the traffic source. Route around via PAC.
🧠
MENTAL NOTE
Two paths from "site unreachable": DNS error (resolution) vs gateway timeout (forwarding). Sequence: verify provisioning → multi-user reproduce → switch to secondary DC. A 504 from only one DC = likely web-server blacklisting; workaround is PAC-route around the impacted node.

💭 THINK FIRST · WRONG PUBLIC SERVICE EDGE

"Why am I being sent to LAX1 when I'm in Atlanta?" sounds like a Zscaler routing bug. Almost always it's the PAC return statement (hardcoded IP, typo), a stale GeoIP entry in MaxMind, or a sub-cloud that doesn't include the geographically nearest PSE.

  • Why is a hardcoded service IP in the PAC return statement the worst kind of misconfig?
Q1 — MODEL ANSWER
A hardcoded IP bypasses Zscaler's GeoIP-based routing entirely. Users in any location will all be forced to the same DC regardless of latency. It also breaks when Zscaler decommissions or moves that IP. The correct return statement uses ${GATEWAY} (or gateway.<cloudname>.net), which lets Zscaler pick the geographically nearest PSE dynamically.
USE CASE 2 · ZIA TRAFFIC FORWARDING

ZIA Traffic Forwarding — Routing to Incorrect Public Service Edge

01 · PROBLEM
Users land at an unexpected Zscaler DC
"Why do I get sent to LAX1 when I'm in Atlanta?" Users connect to a DC much further away than expected.
02 · LOCATE
Pull the PAC file from ZSATunnel logs
1. Locate the PAC URL in ZSATunnel logs (line containing DBG App profile pac type with a path like http://pac.zscalertwo.net/sjDFkosfS/PACFileName).
2. Find the PAC contents in the log via FindProxyForURL or download directly.
3. Review the return statement to see which SE you're sent to.
4. Determine your public IP. Check sub-cloud config at Administration → Sub-cloud.
03 · ISOLATE
Five-step isolation
1. In Admin UI, verify the PAC doesn't have a service IP hardcoded.
2. Verify your public IP geolocates correctly: maxmind.com/en/geoip2-precision-demo?ip_address=.
3. Inspect the return statement for typos.
4. Review sub-cloud to ensure it contains the nearest PSE.
5. Review trust.zscaler.com for issues impacting your nearest DC.
04 · DIAGNOSE
Fix the PAC / IP / sub-cloud
Step 1: if PAC has a hardcoded IP, replace with ${GATEWAY}.
Step 2: if DNS server IP is in a different region or PAC uses gateway.<cloudname>, switch to ${GATEWAY}. Verify MaxMind coords; if incorrect, contact MaxMind.
Step 3: inspect PAC URL in Mobile Admin UI for typos.
Step 3b: if there's a port typo on a service edge return statement, ZCC may default to 443.
Step 4: review sub-cloud — GeoIP resolution only happens within the target sub-cloud, so the standard sub-cloud (gateway.<cloudname>.net / ${GATEWAY}) is the "Public Cloud".
Step 5: check trust.zscaler.com for ongoing primary-DC issues.
🧠
MENTAL NOTE
Wrong PSE → PAC return statement is the #1 cause. Replace hardcoded IPs with ${GATEWAY}. Verify MaxMind coords for your egress IP. Sub-cloud must include the nearest PSE. PAC URL pattern: pac.<cloudname>.net/<hash>/<PACFileName>. gateway.<cloudname>.net = Public Cloud sub-cloud.

💭 THINK FIRST · SLOWNESS

"Slow" is the most ambiguous symptom in the catalogue. Pick one user + one site so you have a stable baseline; then isolate via four narrowing tests before reaching for the heavy diagnostics (MTR, PCAP, MSS/MTU).

  • Why does MTU/MSS matter for performance on Tunnel 2.0, and what's a good first attempt to test fragmentation?
Q1 — MODEL ANSWER
Tunnel 2.0 encapsulates traffic, so encrypted payload + IP/UDP headers can exceed path MTU. When fragmentation happens or PMTUD is blocked by an intermediate firewall, you get heavy retransmissions and visible slowness. A clean test is to move the user to TLS Tunnel 2.0 (TCP fragments more gracefully) or reduce MTU to 1360. If slowness clears, fragmentation was the cause.
USE CASE 3 · ZIA TRAFFIC FORWARDING

ZIA Traffic Forwarding — Slowness Accessing Websites

01 · PROBLEM
Loading is slow through Zscaler
User is connected to the Zscaler service but the website is loading slowly through Zscaler.
02 · LOCATE
Pick one user + one site as a baseline
There are many contributing factors. A logical isolation needs a stable test target — one user, one website — before any data collection.
03 · ISOLATE
Four narrowing tests
1. Identify how many users are affected.
2. Identify if multiple websites are slow.
3. Does the site work without Zscaler or with a different SE?
4. Review web-insights to see Proxy Latency.
04 · DIAGNOSE
DC health · path · packet · support
Step 1: check Zscaler DC health at trust.zscaler.com (Cloud Status).
Step 2: check path or switch over to secondary DC — forward MTR via Z-analyzer to a Zscaler node (IP from ip.zscaler.com), or use speedtest.zscaler.com for reverse MTR. Browse to http://127.0.0.1:9000/test?q=username@domain.com for DNS/UDP reachability, traceroute, throttling, fragmentation, file download direct/ZCC, upload/download bandwidth with/without Zscaler. If clean up to Zscaler, switch to secondary DC.
Step 3: PCAPs + MTU/MSS — take a client PCAP, check for retransmissions; check MTU/MSS on the tunnel interface; on Tunnel 2.0 move user to TLS or reduce MTU to 1360.
Step 4: contact Zscaler Support with all gathered data.
i
FAST DIAGNOSTIC URLs
http://127.0.0.1:9000/test?q=user@domain.com — ZCC Network Test bundle.
ip.zscaler.com — your egress Zscaler node IP.
speedtest.zscaler.com — reverse MTR + throughput.
trust.zscaler.com — Cloud Status.
🧠
MENTAL NOTE
Slowness flow: 1 user + 1 site baseline → trust.zscaler.com → MTR via Z-analyzer → ZCC Network Test (127.0.0.1:9000/test?q=user@domain) → PCAP + MTU. On Tunnel 2.0 fix fragmentation by switching to TLS or reducing MTU to 1360. Don't open a Support case before you've collected this data.

💭 THINK FIRST · ZPA APP CONNECTOR ERRORS

A ZPA app-not-reachable error often reads as a Zscaler problem ("Cannot open connection to server") but the connection that's failing is App-Connector-to-app, not user-to-Zscaler. The diagnostic tools live inside the ZPA Admin UI under Diagnostic → Support Information.

  • Why is the App Connector the right place to run DNS/Ping/TCP tests, rather than the user's machine?
Q1 — MODEL ANSWER
Because the App Connector lives inside the customer's private network alongside the app — that's the segment the connection is actually failing on. The user's machine sits outside the private network; pings from there would test a different path entirely. Running DNS/Ping/TCP from the App Connector reproduces the exact path the failing data session traverses.
USE CASE 4 · ZPA TRAFFIC FORWARDING

ZPA Traffic Forwarding — Application not Reachable

01 · PROBLEM
AC: Cannot open connection to server
Unable to reach Private Access application. ZPA Diagnostic logs show status code "AC: Cannot open connection to server"; internal status code: AST_MT_SETUP_ERR_OPEN_S…; status: Closed Connection.
02 · LOCATE
Read the ZPA Internal Status Code
ZPA Diagnostic logs expose the Internal Status Code. The Help portal lists every ZPA status code and description — match the code first.
03 · ISOLATE
"App Connector → app" connectivity failure
Per the Help "About ZPA Session Status Codes" article, this code means: "The App Connector encountered an error when setting up a data connection to the server." Connection between App Connector and app is the failure surface. Use the built-in diagnostics tools in the ZPA Admin UI.
04 · DIAGNOSE
Run DNS / ICMP / TCP / PCAP from the App Connector
Step 1: DNSLOOKUP of destination server from App Connector via Diagnostic → Support Information.
Step 2: ICMP Ping the destination from the App Connector.
Step 3: TCP Ping the destination from the App Connector.
Step 4: PCAP from the App Connector.
Each test runs from the App Connector, which is exactly where the failing connection originates.
🧠
MENTAL NOTE
ZPA error AC: Cannot open connection to server + internal code AST_MT_SETUP_ERR_OPEN_* = App-Connector-to-app failure. Diagnose path: ZPA Admin UI → Diagnostic → Support Information → DNSLOOKUP → ICMP Ping → TCP Ping → PCAP — all run from the App Connector.

💭 THINK FIRST · FW/AV ERROR

"Endpoint FW/AV Error" looks like an antivirus problem but ZCC actually reports it whenever its health-check probe to 100.64.0.6 fails. That probe can fail because of a missing route, a Windows Firewall block, or a third-party AV interfering with the ZCC process list.

  • What's the role of the 100.64.0.6 probe, and why does ZCC need a default-route to send it through?
Q1 — MODEL ANSWER
ZCC probes 100.64.0.6 to confirm the local firewall and AV aren't blocking ZCC processes. The probe must traverse the user's actual NIC (Wi-Fi or Ethernet), not a VPN or other virtual adapter. If a more-specific route sends it elsewhere, the probe response never comes back and ZCC declares FW/AV Error even though nothing is actually blocking traffic. Adding a specific route for 100.64.0.6 to the default interface forces the probe through the correct path.
USE CASE 5 · ZCC TRAFFIC FORWARDING

ZCC Traffic Forwarding — Endpoint FW/AV Error

01 · PROBLEM
ZCC status: "Endpoint FW/AV Error"
Zscaler Client Connector Connectivity status shows "Endpoint FW/AV Error" + "Off-Trusted Network".
02 · LOCATE
ZCC sends probes to 100.64.0.6
KB article: ZCC sends FW/AV probes on the default NIC to 100.64.0.6. If the probe response doesn't come back, ZCC displays "Endpoint FW/AV Error". The KB also requires that ZCC processes be whitelisted on the local machine.
03 · ISOLATE
Route + Windows Firewall + AV
Route: ensure health-check traffic uses the default route, not another interface. Test with PowerShell: Find-NetRoute -RemoteIPAddress 100.64.0.6 and check the InterfaceAlias.
Windows Firewall: ZCC auto-adds a "Zscaler App Rule" allowing all ports/protocols for Domain/Private/Public. Verify with: netsh advfirewall firewall show rule name = "Zscaler App Rule".
AV interference: Zscaler has whitelisting agreements with some vendors (Trend Micro, Kaspersky). For other endpoint security products, allowlist ZCC processes per the Help portal.
04 · DIAGNOSE
Add route · verify rule · allowlist processes
Route: c:\> route -p add destination_network MASK subnet_mask gateway_ip metric cost
Windows Firewall: ensure ZCC processes from help.zscaler.com/client-connector/zscaler-client-connector-processes-allowlist are bypassed.
AV: allow the same process list at the endpoint security application.
The expected Zscaler App Rule output. If this rule is missing or modified, ZCC probes can be blocked by Windows Firewall.
🧠
MENTAL NOTE
"Endpoint FW/AV Error" = failed probe to 100.64.0.6. Three causes: wrong route, Windows Firewall blocking, third-party AV blocking. Commands: Find-NetRoute -RemoteIPAddress 100.64.0.6, netsh advfirewall firewall show rule name = "Zscaler App Rule". Allowlist source: help.zscaler.com/client-connector/zscaler-client-connector-processes-allowlist.

💭 THINK FIRST · CAPTIVE PORTAL DETECTION

ZCC detects captive portals with two probes: a generate_204 request expecting HTTP 204, and a PAC-file download from pac.<cloudname>.net. If either returns anything other than what's expected, ZCC declares "Captive Portal Detected". False positives happen when those endpoints are unreachable for a non-captive reason.

  • Why does the generate_204 trick reliably detect captive portals?
Q1 — MODEL ANSWER
A "204 No Content" response has no body, so the only thing it carries is its status code. A captive portal can't fake a 204 — by definition it has to redirect (302/307) to its sign-in page or serve a different response. So if ZCC asks for a 204 and gets anything else, that's almost always proof of an intercepting middlebox between ZCC and the internet.
USE CASE 6 · ZCC TRAFFIC FORWARDING

ZCC Traffic Forwarding — Captive Portal Error

ZCC's "Captive Portal Detected" status. True or false positive — the ZSATunnel logs tell you which.
01 · PROBLEM
ZCC shows "Captive Portal Detected"
User on a network (e.g. airport Wi-Fi, hotel, coffee shop) — but sometimes on the corporate network — sees the captive-portal warning in ZCC.
02 · LOCATE
Two detection mechanisms
Probe 1: ZCC reaches out to http://gateway.<cloudname>.net/generate_204 expecting HTTP 204. Anything else triggers a captive portal flag.
Probe 2: ZCC downloads the default PAC file from http://pac.<cloudname>.net/<cloudname>.net/proxy.pac. Download failure triggers a captive portal flag.
03 · ISOLATE
Review ZSATunnel logs · search ZCPM
Export ZCC logs and search for ZCPM (Zscaler Captive Portal Manager) entries. Sample failing log:
DBG ZCPM detectCaptive: Connecting to url: http://gateway...
DBG ZCPM detectCaptive: Response Status 302 Length…
302 instead of 204 means a middlebox is intercepting — captive portal detection is correct.
04 · DIAGNOSE
Restart · validate Gateway/PAC URLs reachable
1. Restart ZCC to confirm if the issue clears.
2. Check where the user is — if it's a public Wi-Fi the captive portal is real; the user just needs to sign in.
3. Ensure http://gateway.<cloudname>.net/generate_204 and http://pac.<cloudname>.net/<cloudname>.net/proxy.pac are reachable from the user's machine with the expected HTTP response codes (204 and 200 respectively).
ZSATunnel logs with the ZCPM probe receiving a 302 instead of 204 — confirms an intercepting captive portal.
🧠
MENTAL NOTE
Captive Portal Detected = 1 of 2 probes failed: gateway.<cloudname>.net/generate_204 (expects 204) or PAC download from pac.<cloudname>.net (expects 200). Search ZCPM in ZSATunnel logs. Restart ZCC, verify probe reachability from the user's network. Real captive portals on hotel/airport Wi-Fi just need user sign-in.

💭 THINK FIRST · PERFORMANCE QUICK REFERENCE

For performance complaints, the course-provided Performance Issues quick-reference table splits diagnostics across four axes: Through Zscaler, Specific to DC, Specific to Location/Network, and Specific to User/Group. The matrix is the fastest "what do I check next?" lookup for performance tickets.

  • Why is "test the site without Zscaler" always the first row in every column of this matrix?
Q1 — MODEL ANSWER
Because if the site is also slow without Zscaler in the path, the issue can't be Zscaler. That single test eliminates Zscaler from the suspect list and saves you from instrumenting tunnels, sub-clouds, and proxy latency for a destination problem. It's the cheapest, fastest first-pass test in performance triage.

Things to remember

Six common gotchas for performance troubleshooting — speedtest accuracy with proxy, PAC dnsResolve, IPSec PFS overhead, GRE/IPSec encapsulation overhead.
i
SIX GOTCHAS TO INTERNALISE
1. Generic network speed tools may not be accurate when tested through a proxy.
2. A reliable way to measure speed is to download a sample 100MB file and measure speed using Task Manager or Wireshark.
3. Slowness can also be due to browser issues — clear cache, exceptions, advanced settings, addons/plugins, frequent 403s.
4. In PAC files, use Best Practices for internal DNS resolution (dnsResolve()). Also applies to App-profile + Forwarding profile PAC.
5. IPSec tunnels with PFS add overhead and can cause performance issues. Disable PFS where possible.
6. Location-based performance issues with supported throughput on GRE/IPSec traffic — encapsulation/decapsulation often spikes resource utilisation.
🧠
MENTAL NOTE
Two performance matrices: ZIA (columns: Through Zscaler / DC / Location / User) and ZPA (columns: Website-App / Location / Single User). Six gotchas: speedtest through proxy, 100MB sample test, browser-side slowness (cache, 403s), PAC dnsResolve() best practices, IPSec PFS overhead, GRE/IPSec encapsulation overhead.
