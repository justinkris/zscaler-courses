Troubleshoot and Diagnose Branch Issues

Gateway CLI access and core commands, DHCP and routing validation, ZIA IPSec/GRE diagnostics, HA switchover, VRRP verification, and WAN link health monitoring.

CLI first, assumptions last — evidence-based troubleshooting at the branch edge.

Zero Trust Branch issues can span routing, DHCP, HA failover, ZIA tunnel health, and container services all at once. This module arms you with the CLI commands and systematic workflows to isolate root causes quickly — from a simple ping to a full VRRP state inspection.

MODULE OVERVIEW

Systematic diagnosis from the gateway CLI

This module covers how to troubleshoot common and advanced Zero Trust Branch operational issues using gateway CLI workflows and targeted checks for routing, DHCP, tunnel health, and high availability — so you can quickly isolate the root cause and restore service.

Learning Objectives

  • Access the Airgap Gateway Platform CLI and execute core network diagnostic commands (ping, traceroute, route inspection)
  • Capture and interpret packet-level evidence using monitor/tcpdump to validate DHCP and other control-plane flows
  • Validate DHCP relay behavior by confirming routing and bidirectional reachability from the LAN subnet to the DHCP server
  • Diagnose ZIA integration issues by verifying IPSec/GRE tunnel configuration, PBR marking, and related services (e.g., StrongSwan)
  • Access high-availability and link-health behavior by initiating switchover and reviewing VRRP/WAN monitoring signals

Three Troubleshooting Areas

General CLI Troubleshooting
Platform CLI access, common network/Linux commands, traffic capture with monitor and tcpdump.
DHCP Troubleshooting
DHCP relay validation, packet-level verification of DORA handshake over routed tunnel.
Advanced Network Diagnostics
HA switchover, ZIA IPSec/GRE, PBR flow marking, ZPA containers, VRRP, and WAN monitoring.
Module Outcome
You'll be equipped to deploy and support a Zero Trust Branch environment with confidence using CLI evidence.

Essential Diagnostic Commands

1
ping — Test basic reachability
Sends ICMP echo requests to a specified IP or hostname. Used to confirm an interface can reach devices on its segment.
2
traceroute — Map the network path
Reveals each router hop between the gateway and the destination. Helps identify where packets are being forwarded or dropped, and confirms whether traffic follows the expected WAN or ZPA path.
3
show ip route — Inspect the routing table
Displays all active routes including WAN, LAN, ZPA tunnels, and IPsec entries. Confirms the gateway knows how to forward traffic to all expected destinations.
4
monitor traffic interface GE8 filter icmp — Platform capture
Built-in platform command for packet capture. Applies a filter so only ICMP traffic is shown. Confirms whether traffic generated by ping is actually seen on the specified interface.
5
tcpdump -i GE8 icmp — Linux-level cross-validation
Low-level Linux packet capture utility. Cross-validates the monitor traffic output and provides raw packet-level detail. Both tools confirming the same traffic gives high confidence in the result.
# Test connectivity to a host ping 192.168.8.1 # Trace the network path to confirm routing traceroute 8.8.8.8 # View all active routes show ip route # Platform-native capture on GE8 (ICMP only) monitor traffic interface GE8 filter icmp # Linux-level cross-validation tcpdump -i GE8 icmp
Key Takeaway: Both monitor traffic and tcpdump confirm live ICMP traffic on GE8. Using both tools cross-validates your capture and deepens CLI confidence on ZT Branch appliances.

HA & VRRP TROUBLESHOOTING

Initiate Switchover and verify VRRP state

Initiate Switchover

In a Zero Trust Branch architecture, Initiate Switchover is a mechanism that swaps the roles of the Master (Active) and Backup (Standby) gateways. This ensures seamless failover and allows administrators to troubleshoot without disrupting operations.

Validate Failover Behavior
Confirm the Standby can take over cleanly before a real failure occurs. Used during commissioning and after changes.
Upgrade Testing Without Downtime
Update or modify the Standby gateway, then initiate a switchover to validate changes under real traffic conditions while keeping redundancy intact.

WAN Link Quality Monitoring

The ZTB Agent Module continuously monitors WAN link health and steers traffic to the best available link:

1
Discover nearest ZIA proxy VIP
ZTB Agent sends a GET request to https://gateway.<zcloud-name>/get_loc to discover the nearest ZIA proxy VIP. This ensures TCP probes are directed to the closest Zscaler gateway, minimizing baseline latency.
2
Evaluate link quality (jitter, latency, packet loss)
TCP probes measure link quality metrics. Scores are normalized to produce the 0–30 Link Score seen in the Interface Statistics dashboard.
3
Select best WAN link
Traffic is steered into a dedicated "best" routing table where the healthiest link is given the lowest metric. The lowest-metric default route becomes the path of least resistance.

ZIA & ADVANCED DIAGNOSTICS

ZIA IPSec/GRE tunnels, PBR marking, and containers

ZIA IPSec Troubleshooting — 5-Step Flow

1
swanctl.conf sanity
Confirm addresses and proposals in the StrongSwan configuration file are correct.
2
Find the fwmark
Identify the firewall mark that selects the IPSec table for the relevant traffic class.
3
Confirm zia_ipsec_rt
Routes in the ZIA IPSec routing table should point to ipsec0/ipsec1 interfaces.
4
Prove MARK is applied
The iptables MARK value must match the fwmark — confirms policy routing is engaged.
5
Validate selectors
Confirm source and destination selector sets are correctly defined for the tunnel.
ZIA IPSec activation flow: Upon Gateway activation, the Poller pulls IPSec credentials from the ZIA API, discovers nearby ZIA IPSec endpoints, and automatically generates the swanctl.conf configuration file needed by StrongSwan.

ZIA GRE Troubleshooting — PBR Steering

Client packets matching PBR rules are redirected to za-gre-route-table and forwarded to the Kernel. Policy-based routing evaluates each packet and steers qualifying traffic toward the GRE path instead of the default route.

i
MULTI-WAN STATIC ROUTE ENFORCEMENT
In multi-WAN setups, static routes ensure GRE packets egress via the correct source address interface. This prevents asymmetric routing where GRE-encapsulated packets might leave through one WAN link while responses return via a different interface — which would break the tunnel.

Debug ZIA over IPSec — Key Check

When ZIA forwarding is configured over IPSec but users intermittently bypass it, validate PBR flow marking on the gateway console to confirm traffic is matching the policy and being redirected to the IPSec interface.

A branch is configured for ZIA forwarding over IPSec, but users intermittently appear to bypass ZIA. Which check most directly confirms whether the gateway is steering the intended client traffic into the IPSec tunnel?
✕
Initiate a switchover and assume the new Active gateway will automatically correct any steering problems
✓
Validate Policy-Based Routing (PBR) flow marking on the gateway console to confirm the policy is hit and traffic is redirected into the IPSec interface
✕
Recreate the ZIA location and VPN credentials first, because bypass behavior always indicates bad credentials
✕
Focus only on tunnel encryption/decryption behavior in the kernel, because steering happens after traffic is encrypted
