Integrate Third-Party Security Platforms

CrowdStrike posture-based access, Armis asset intelligence for microsegmentation, and SIEM log export for centralized monitoring and compliance.

Posture, intelligence, and logs — Zero Trust extended to the entire security ecosystem.

Zero Trust Branch doesn't operate in isolation. By integrating CrowdStrike, Armis, and a SIEM, ZTB gains real-time device health scores, agentless asset classification, and centralized event correlation — enabling autonomous, context-aware policy enforcement across every device type.

MODULE OVERVIEW

Enriching Zero Trust Branch with third-party security context

This module covers three integrations that extend Zero Trust Branch beyond network-level security: CrowdStrike for real-time endpoint posture scoring, Armis for agentless asset intelligence, and SIEM for centralized log collection and correlation.

Learning Objectives

  • Configure and validate the CrowdStrike integration in Zero Trust Branch
  • Analyze device posture scores to build device objects and apply posture-based segmentation policies
  • Configure and validate the Armis integration and verify learned intelligence on assets
  • Configure and validate SIEM log export and confirm log receipt on the SIEM server
CrowdStrike Integration
Configure and validate integrations with CrowdStrike Falcon to extend ZTB with real-time endpoint posture and health information.
Posture-Based Policies
Use CrowdStrike ZTA posture scores to build device groupings and strengthen segmentation and access decisions automatically.
Armis Asset Intelligence
Integrate Armis to discover, classify, and continuously assess risk across all IT, IoT, and OT devices — managed and unmanaged.
SIEM Log Export
Forward Zero Trust Branch logs to a SIEM to support centralized visibility, investigation workflows, and compliance reporting.

Real-time endpoint posture for conditional access

Configuration Steps

Portal path: Infrastructure › Connectors › Edge › Integrations › CrowdStrike Integration Settings
1
Generate API keys in CrowdStrike
In the CrowdStrike portal, generate API keys (Client ID, Client Secret, Base URL) for the ZTB integration.
2
Enter credentials in ZTB
In ZTB, enter the Client ID, Client Secret, and Base URL from the generated API keys.
3
Run a connectivity test
Run a quick connectivity test to verify the credentials are correct and the integration can reach the CrowdStrike cloud.
4
Save to activate
Save the configuration to activate the integration. ZTB will now continuously pull ZTA scores from CrowdStrike via API.
5
Install Falcon Sensor on endpoints
Endpoints must have the Falcon sensor installed before the integration takes effect. Download from the CrowdStrike console, select the correct OS, and install. Once running, endpoints appear in CrowdStrike under Host Management.

What ZTB Continuously Pulls from CrowdStrike

Device Details
Device identity, manufacturer, model, and category information for accurate asset classification.
OS Information
Operating system version, patch level, and configuration details used in posture scoring.
Patches Installed
Current patch state of the endpoint — missing patches lower the ZTA score immediately.
ZTA Score
The Zero Trust Assessment score (0–100) reflecting overall device health and security posture in real time.

Key Takeaways

  • API Integration: ZTB and CrowdStrike communicate via APIs to continuously exchange device posture and health information
  • Falcon Sensor: The foundation for collecting endpoint telemetry used for posture scoring — must be installed on all managed endpoints
  • Device Objects and Policies: ZTB uses posture scores to classify devices into risk tiers and applies the appropriate segmentation policy automatically
  • End-to-End Verification: ZTB asset posture scores confirmed to match CrowdStrike ZTA scores, and policy enforcement aligned with expected access behavior
  • Continuous Posture-Based Access: ZTB and CrowdStrike together deliver real-time, posture-driven access control that strengthens security without compromising user experience

ARMIS INTEGRATION

Agentless asset intelligence for dynamic microsegmentation

Armis is an agentless cyber asset intelligence and security platform that discovers, classifies, and continuously assesses risk across connected devices — managed and unmanaged — in enterprise environments including IT, IoT, OT, and medical devices.

How Armis Intelligence Drives Conditional Access Policies

ZTB leverages Armis device intelligence such as asset classification, security tags, and risk insights to enforce dynamic, conditional access policies:

Device Status (Armis)Access Policy
Managed Corporate + CompliantAllow full application and network access
Unmanaged or Unknown (missing ownership/controls)Allow limited access to approved apps and services only
High Risk, Vulnerable, or Under AttackBlock access or quarantine to a restricted segment
✓
KEY CONCEPT
Armis risk score thresholds drive autonomous policy enforcement. Devices tagged High Risk, Vulnerable, or Under Attack are blocked or quarantined, while Unmanaged/Unknown devices receive limited access, and Managed Corporate compliant devices receive full access.

Configuration

Zscaler side: API-based integration configured in the Zscaler platform. An API key provisioned from the Armis platform is entered into ZTB to enable secure data ingestion.

Armis side: Integration also configured within Armis, allowing Armis to ingest asset data from Zscaler. This bidirectional exchange provides enriched context to both platforms.

Assets tagged with the "Armis" prefix in Zscaler confirm the integration is actively enriching device data in real time.

What Armis Brings to ZTB (CorTech Manufacturing Use Case)

Armis + ZTB was deployed at CorTech Manufacturing (40,000+ employees, multiple continents) to protect all IT, IoT, and OT assets enterprise-wide.

Real-Time Discovery & Monitoring
Armis discovers and monitors all connected assets — IT, IoT, OT, medical devices — eliminating security blind spots across the entire attack surface. No agents required.
Continuous Risk Assessment
Armis continuously assesses risk exposure and enables automated, proactive remediation to reduce cyber risk and contain threats before they escalate.
Enriched Asset Discovery
JA3 hashes (TLS client fingerprints), Fing network scanning (device identification), and Armis data annotations provide rich contextual device details.
Seamless Integration
Integrates with other security tools for unified cyber exposure management. ZTB dashboard shows discovered assets by device type: automation, handheld, imaging, manufacturing, medical, network.

SIEM INTEGRATION

Centralized log collection, correlation, and compliance

A Security Information and Event Management (SIEM) system is a centralized security platform that collects, correlates, and analyzes logs and alerts across systems to detect threats, support investigations, and meet compliance needs.

Why SIEM Integration Matters

Consolidate Events in One Place
View ZTB logs alongside other security telemetry to get a unified picture of all network activity — no tool switching required.
Correlate Activities Across Systems
Tie ZTB events to identities, devices, and network indicators to validate suspicious behavior faster and detect abnormal patterns.
Respond to Incidents Efficiently
Use SIEM alerting, cases, and automation to streamline response and reduce time to containment.
Retention & Compliance
Store logs for the timeframe required by policy. Preserve searchable event history to support audits and incident reviews.

Configure SIEM Integration in ZTB

Portal path: Infrastructure › Connectors › Edge › Integrations › SIEM Integration › Settings
1
Log in to the Experience Center portal
Navigate to the SIEM Integration settings.
2
Enter SIEM Server details
SIEM Host: <SIEM Server IP> · SIEM Protocol: TCP · SIEM Port: 514 (default for SIEM reception)
3
Click Confirm
Confirm the configuration entries before proceeding.
4
Run a validation test
Run a test to validate the integration and look for confirmation once validated.
5
Click Save
Save the configuration to activate log forwarding to the SIEM.

Validate the SIEM Server (Ubuntu / rsyslog)

Verify that the rsyslog service is running on the SIEM server. Run each command in sequence:

sudo apt update sudo apt install rsyslog -y sudo systemctl enable rsyslog sudo systemctl start rsyslog sudo systemctl status rsyslog

ZTB Policy Configuration for rsyslog

Configure a catch-all firewall policy (Allow-All-rsyslog) to allow all traffic to ensure logs can flow to the SIEM server.

Validate SIEM Logs

Once SIEM integration is enabled, verify logs are received on the rsyslog server. Default system logs are stored in:

/var/log/syslog
