COURSE OVERVIEW

The big picture in four sentences

Zero Trust Branch replaces legacy hub-and-spoke networking with an inline gateway that puts every device on its own /32 network, routes all traffic through Zscaler's cloud for inspection, and enforces identity- and context-based policies at the branch edge — with no implicit trust anywhere.

You deploy it by creating a Site, activating an appliance, and it phones home. You segment with /32 airgap. You integrate ZIA for internet, ZPA for private apps, CrowdStrike/Armis for posture. You monitor with Analytics and troubleshoot via the Gateway CLI.

💭 THINK FIRST · SECTION 1 — ZERO TRUST CONCEPTS

Networks have traditionally operated on a "castle and moat" model — once you're inside the perimeter, everything trusts you. That worked when users, devices, and apps stayed in one building. It doesn't hold when the branch edge is a gateway to everywhere.

  • If a laptop inside the branch gets compromised, what stops it from attacking every other device on the same network?
  • Why does being "inside the network" no longer equal being trustworthy — especially at a branch site?
  • If you had to redesign branch networking with zero default trust, where would you place the policy enforcement point — and why?
Q1 — MODEL ANSWER
On a traditional flat /24 branch LAN, nothing stops lateral movement — once a laptop is compromised, the attacker can scan and pivot freely because the segment trusts itself. ZTB closes this by making the inline gateway the only path between any two devices, so east-west traffic is policy-checked just like north-south.
Q2 — MODEL ANSWER
Branches now host IoT, contractor laptops, BYOD, and M&A devices alongside corporate endpoints, so "inside" is no longer a meaningful trust boundary. Zero Trust shifts trust evaluation to identity, device posture, and context — evaluated continuously by the Zero Trust Exchange — rather than to physical location.
Q3 — MODEL ANSWER
The enforcement point belongs inline at the branch gateway itself, where every packet must traverse it before reaching any other device or the internet. That is exactly what ZTB does: it installs as the inline gateway and pairs with ZTE in the cloud, so policy follows the user and device rather than the network segment.

💭 THINK FIRST · SECTION 2 — SEGMENTATION

On a traditional /24 subnet, any device can reach any other device on that segment without going through a gateway. Now consider what happens if a printer, a surveillance camera, and a finance laptop all share that subnet — and one of them gets infected.

  • What changes about device communication when every device has a /32 instead of a shared /24 — what must now happen for any two devices to talk?
  • Which category of attack becomes nearly impossible when east-west traffic always flows through the gateway?
  • Why would a hospital or a manufacturer care about IoT isolation more intensely than a typical office network would?
Q1 — MODEL ANSWER
With a /32 mask, every device is its own "segment of one" — there are no neighbors in its broadcast domain. Any communication between two devices has to be routed through the ZTB gateway, where a firewall or segmentation policy decides whether it is allowed.
Q2 — MODEL ANSWER
Lateral movement (east-west propagation) becomes nearly impossible because no device can reach another without an explicit permit at the gateway. Worms, ransomware, and recon scans that rely on ARP discovery and same-subnet pivots are effectively neutralized.
Q3 — MODEL ANSWER
Hospitals and factories run unpatched, long-lifecycle devices — infusion pumps, MRIs, PLCs, robots — that cannot host an EDR agent and where downtime is unsafe. Airgap /32 isolation contains a compromised IoT/OT device automatically, protecting patient-care or production-floor systems from a flat-network blast radius.

💭 THINK FIRST · SECTION 3 — DEPLOYMENT

A new gateway arrives at a branch site. It has no configuration yet. Before it can do anything useful, it needs to "phone home" and receive its instructions from the cloud. Think carefully about the direction of that connection — who initiates it, and why that matters.

  • Does Zscaler reach into the branch to configure the appliance, or does the appliance reach out to Zscaler? What are the firewall implications of each direction?
  • SDA (browser-based) is the preferred method, but a serial console option also exists — under what conditions would you actually need the console?
  • If activation requires port 443 outbound and UDP 500/4500 — and your branch firewall blocks all outbound traffic by default — what do you need to do before the appliance can activate?
Q1 — MODEL ANSWER
The appliance always reaches outbound to Zscaler — Zscaler never reaches inbound. That means no inbound ports are exposed at the branch firewall and no inbound NAT or DMZ is required; only outbound TCP 443 (and IPSec UDP 500/4500 once tunnels come up) is needed, which keeps the branch attack surface near zero.
Q2 — MODEL ANSWER
Use the serial console (PuTTY at 115200 baud, options 4 and 6) when SDA can't be used — DHCP is unavailable on the WAN interface, the activation URL is unreachable, you need to pre-stage networking before plugging in, or you must collect diagnostics on a stuck activation. It is also handy for verifying interface roles and reading platform CLI output.
Q3 — MODEL ANSWER
You have to pre-authorize outbound flows on the upstream firewall before plugging the appliance in: TCP 443 to the Zscaler activation/cloud endpoints, plus UDP 500 and UDP 4500 for IKE/IPSec to the assigned ZIA data center. Without those allowed, the appliance will not call home and SDA will hang.

💭 THINK FIRST · SECTION 4 — TUNNELS & INTEGRATION

All branch internet traffic is about to be routed through a cloud security service. Every packet takes a detour — branch → tunnel → Zscaler cloud → internet. That's a deliberate design choice. Think about what you gain and what overhead you accept.

  • GRE is faster than IPSec but doesn't encrypt. IPSec encrypts but adds overhead. When would the trade-off favor one over the other?
  • What can a cloud security service like ZIA inspect that a traditional hardware firewall sitting at the branch edge could not?
  • IKEv1 and IKEv2 both handle key exchange for IPSec — what specific problem does IKEv2 improve on, and why does it matter in a branch deployment?
Q1 — MODEL ANSWER
IPSec is the default for any traffic crossing the public internet because it provides confidentiality, integrity, and authentication. GRE makes sense when the underlying link is already trusted or encrypted (private MPLS, dark fiber, or with IPSec layered on top) and you need lower CPU overhead or to carry multicast/non-IP protocols that pure IPSec cannot.
Q2 — MODEL ANSWER
ZIA in the cloud can do TLS/SSL inspection at scale, full URL and cloud-app categorization, sandboxing of unknown files, DLP, and threat intelligence correlated across every Zscaler customer — without the branch having to refresh hardware. A box at the branch is limited by its appliance CPU, signature feed, and the volume of traffic it can decrypt locally.
Q3 — MODEL ANSWER
IKEv2 fixes IKEv1's fragility on flaky branch WAN links: fewer messages to establish the SA, built-in NAT traversal and dead-peer detection, MOBIKE for IP changes, and EAP authentication. The practical impact at a branch is faster tunnel setup, cleaner re-keying, and tunnels that survive ISP blips instead of dropping user sessions.

💭 THINK FIRST · SECTION 5 — ADMIN & POLICIES

You manage 80 branch sites and need to push a new firewall policy. You could touch each site individually — or you could think about how template systems work in other parts of IT. But templates don't cover everything. Some rules need to be site-specific.

  • If firewall rules evaluate Site → Template → Global → System and "first match wins" — where does a site-specific exception need to live to take effect before a template rule blocks it?
  • What happens to a packet that matches none of the rules at any level? Is that safe?
  • The Ransomware Kill Switch isolates the entire network with one click. Under what circumstances is that the right call — and what do you have to accept as the trade-off?
Q1 — MODEL ANSWER
The exception has to be created at the Site level, because Site is evaluated first in the Site → Template → Global → System order and first match wins. If the exception lives in the Template, the template's own block rule above it (or the surrounding template logic) will hit first and the override never gets a chance.
Q2 — MODEL ANSWER
A packet that matches no Site, Template, or Global rule falls through to the System Default, which is an implicit deny. That is safe (fail-closed) — nothing accidentally allowed — but it also means you must explicitly permit anything you actually want to flow, including required management or monitoring traffic.
Q3 — MODEL ANSWER
RKS is the right call when a confirmed ransomware or worm outbreak is spreading and containment beats availability — for example, multiple endpoints encrypting files or beaconing to known C2. The trade-off is total user impact: phones, POS, printers, and clinical or production devices all lose connectivity, so it should only be triggered with incident-response authority and a recovery plan ready (Green → Yellow → Orange → Red).

💭 THINK FIRST · SECTION 6 — ADVANCED NETWORK SERVICES

A single gateway is a single point of failure. If it reboots mid-day, every device at the branch loses connectivity. High availability solves this — but seamless failover requires more than just a second device. Think about what the backup needs to know for active sessions to survive a switchover.

  • If the Master gateway fails mid-TCP-session, what data must the Backup already have in order for the user to not notice the switchover?
  • SNMPv2 uses a "community string" — essentially a shared password in plaintext. What exactly does SNMPv3 add, and what threat does each addition mitigate?
  • DNS Override lets you rewrite DNS responses at the branch. What legitimate use case justifies this — and what could go catastrophically wrong if it's misconfigured?
Q1 — MODEL ANSWER
The Backup needs the full connection-tracking (Conntrack) state — the active 5-tuples, sequence numbers, NAT mappings, and tunnel SAs — so when VRRP fails over and the virtual IP plus a gratuitous ARP move, in-flight TCP sessions continue without reset. That is exactly what the VRRP HA link (dedicated in v7.8+) and Keepalived/Conntrack sync provide.
Q2 — MODEL ANSWER
SNMPv3 adds authentication (HMAC-SHA), which prevents a spoofed poller from impersonating the NMS, and privacy (AES/DES encryption), which prevents passive sniffing of OIDs and counters on the wire. Together they replace the v2 model where anyone who learns the community string can both read and forge SNMP traffic.
Q3 — MODEL ANSWER
Legitimate uses include pinning an internal hostname to a private IP at a branch, redirecting captive-portal or guest DNS, or blackholing known malicious domains locally. Misconfigured, it can silently send legitimate traffic (banking, SaaS, internal apps) to the wrong address — effectively a man-in-the-middle by the branch admin — which is why Override should be scoped tightly and reviewed.

💭 THINK FIRST · SECTION 7 — APP CONNECTOR & ROUTING

Traditional VPN creates an inbound listener — a port the firewall must allow inbound, visible to port scanners and attackers. App Connector eliminates that entirely. Think about what it means for your attack surface when zero inbound ports need to be opened on the data center firewall.

  • If App Connector only makes outbound connections and never listens inbound, how does a user's request actually reach the private application?
  • BGP learns routes dynamically and adapts when links go up or down. What operational problem does this solve compared to maintaining static routes across dozens of branch sites?
  • Routed tunnels preserve the original source IP. Which types of applications specifically break if the source IP changes mid-session — and what do they have in common?
Q1 — MODEL ANSWER
Both the user (via ZCC or ZTB) and the App Connector make independent outbound connections to the Zero Trust Exchange. ZTE authenticates and authorizes the request, then stitches the two outbound sessions together — so traffic reaches the private app without ever opening an inbound port on the data center firewall.
Q2 — MODEL ANSWER
Static routes don't scale: every link change, new subnet, or M&A site requires manual updates across dozens of devices, and a missed entry creates a blackhole. BGP advertises prefixes dynamically with attributes like local-pref and AS-path, so failover, new subnets, and policy changes propagate automatically — turning a manual config exercise into a peering relationship.
Q3 — MODEL ANSWER
VoIP signaling, TACACS+/RADIUS, ATM/banking transactions, and Active FTP all break when the source IP is rewritten. They share a common trait: the server enforces ACLs based on the real client IP or embeds that IP inside the application payload (SDP, port-command, callback), so a NAT mid-path corrupts the session.

💭 THINK FIRST · SECTION 8 — MONITORING & ANALYTICS

You can collect all traffic data, or you can collect the right data. Traffic Flow Charts, link scores, packet logs, and flow logs each give you a different view of what's happening on the network. Think about which tool you'd reach for in different situations — and what each one cannot tell you.

  • You suspect two devices that should never communicate are talking to each other. Which monitoring tool surfaces this, and how is the data represented?
  • A branch user reports slow internet. The link score is 4 out of 30. What does that number tell you, and what does it not tell you about the root cause?
  • Flow logs record one entry per 4-tuple per hour by default. Packet logs capture every individual packet. When would the detail of packet-level capture be worth the storage cost — and when would it be overkill?
Q1 — MODEL ANSWER
The Traffic Flow Chart surfaces it: built on a graph database, each device is a node and each observed conversation is an edge between them. An unexpected edge between two nodes — for example a camera talking to a finance laptop — is visually obvious. For near-real-time confirmation use Flow Logs, because the chart can lag by up to ten minutes.
Q2 — MODEL ANSWER
A link score of 4/30 means the WAN link is in poor health, normalized from latency, jitter, and packet loss over the last 30 iterations. It doesn't tell you which of those three metrics is driving it or whether the cause is upstream ISP, local cabling, or congestion — for that you drill into Sites → Interfaces → Interface Statistics 24h.
Q3 — MODEL ANSWER
Packet logs are worth the cost for active incident response, deep protocol or TLS handshake analysis, malware C2 investigation, or proving a specific failure mode to a vendor — anything where headers and payload bytes matter. They are overkill for capacity planning, top-talkers, or "who talked to whom" trending, where flow logs (one entry per 4-tuple per hour) give the same answer at a fraction of the storage.

💭 THINK FIRST · SECTION 9 — THIRD-PARTY INTEGRATIONS

Device trust isn't a binary yes/no — it's a spectrum that changes over time. A fully-patched laptop with active security software is more trustworthy than one that hasn't been updated in 90 days. Now think about what it would take to automatically enforce network access policies based on that continuously-updating score.

  • A user with a ZTA score of 80 is in the middle of a critical meeting. Their score drops to 20 because a patch check failed. What should happen to their network access — and is there any way to make that policy more nuanced than a hard cut?
  • Armis discovers and classifies devices without installing any agent. What signals could it possibly use to identify whether a device is a medical infusion pump versus a security camera?
  • Why is correlating ZTB logs in a SIEM more powerful than reviewing them inside the ZTB portal alone?
Q1 — MODEL ANSWER
Per the standard ZTA tiers, a score of 20 (<30) means the device should be blocked, but a hard cut mid-meeting is disruptive. A more nuanced policy uses tiered enforcement: >70 full access, 30–70 restricted (for example ICMP plus an allowlist for collaboration apps), <30 quarantine — combined with user notification and a "remediate within X minutes" grace window so the user can patch and recover access without an abrupt drop.
Q2 — MODEL ANSWER
Armis fingerprints devices passively from network metadata: MAC OUI, DHCP options, mDNS/SSDP advertisements, TLS/JA3, HTTP user-agents, open ports, and the destinations they talk to. It then matches that behavioral fingerprint against a crowdsourced knowledge base — so an infusion pump's vendor cloud and protocol pattern look very different from an ONVIF camera's, even without an agent.
Q3 — MODEL ANSWER
A SIEM correlates ZTB events with identity (IdP), EDR (CrowdStrike), email, and cloud logs, so a single suspicious flow at the branch can be tied to a user login from a new country and an EDR detection on the same host. The portal alone shows ZTB's slice; the SIEM provides cross-source context, long-term retention, and compliance-grade search that turn isolated events into a story.

💭 THINK FIRST · SECTION 10 — TROUBLESHOOTING

Troubleshooting is a process of elimination — you start broad and narrow down. The temptation is to jump to conclusions ("it must be the firewall") instead of building evidence first. Think about the most logical sequence of checks, from widest scope to most specific.

  • Ping succeeds to 8.8.8.8 but the web application won't load. What does that tell you about where the problem is — and what does it rule out?
  • show ip route and traceroute both help you understand paths. What does one show that the other doesn't, and when would you need both?
  • ZIA IPSec tunnels show as ESTABLISHED but branch traffic still isn't reaching the internet. The problem is post-tunnel. What would your next three diagnostic steps be?
Q1 — MODEL ANSWER
A successful ping to 8.8.8.8 proves Layer 3 reachability, the default route, and NAT/tunnel egress are all working. It rules out a dead WAN, a broken default route, and a fully-down IPSec tunnel. The problem is higher in the stack — DNS resolution, TLS inspection, URL filtering, application policy, or the destination itself.
Q2 — MODEL ANSWER
show ip route shows the control-plane view: which next-hop the router intends to use for a destination, along with the routing protocol and metric. traceroute shows the data-plane reality: the actual hops a packet traverses right now, including asymmetric paths or mid-path drops. You need both when the routing table looks correct but traffic still fails — the table tells you intent, the trace tells you what's really happening.
Q3 — MODEL ANSWER
First, run tcpdump on the WAN/tunnel interface to confirm packets are actually entering the tunnel after PBR. Second, check PBR flow marking and the routing decision (ZIA vs ZPA vs direct) to make sure the intended traffic is being steered into the tunnel. Third, validate on the ZIA side — Location is attached, GRE/IPSec status is up at the data center, and the public source IP matches the configured Location — then confirm there is no upstream firewall rule blocking egress from the tunnel egress IP.
