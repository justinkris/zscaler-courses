IPSEC TUNNEL

Understanding IPSec

Internet Protocol Security (IPSec) is a suite of protocols that provides network-layer security to a Virtual Private Network (VPN). A VPN is a virtual network connection that provides a secure communication path between two peers on a public network.

Peers in an IPSec VPN can be: two hosts (host-to-host), a remote host and a network gateway, or the gateways of two networks — for example, the gateway of your corporate network and a ZIA Public Service Edge.

Three Key Protections

CONFIDENTIALITY
Protect Sensitive Data
Prevents unauthorised access to data. Encryption protects data in transit and at rest. Access controls limit who can view or modify information. Data classification identifies which data needs strongest protection.
INTEGRITY
Detect and Prevent Tampering
Ensures data remains accurate and unaltered throughout its lifecycle. Hashing algorithms verify data has not been modified. Digital signatures prove origin. Audit trails record all changes.
AUTHENTICATION
Verify Identity
Verifies identity of users, devices, or systems before granting access. MFA combines something you know, have, or are. Zero Trust requires continuous authentication — not just at the perimeter.

IKE — Internet Key Exchange

IPSec peers use a negotiation process called Internet Key Exchange (IKE) to define the security mechanisms they will use. There are two versions:

  • IKEv1 — original version; still supported in many deployments
  • IKEv2 — modern, faster, more reliable; recommended for new deployments
  • Authentication methods: Pre-Shared Key (PSK) or X-Auth — both peers must match the same credential method for successful tunnel establishment

ZIA Integration via IPSec

By default, Zero Trust Branch establishes IPSec tunnels on the WAN side to route internet-bound traffic through ZIA's Security Cloud, where it is inspected for malware, threats, and policy compliance.

⚙
Simplified Deployment
With IPSec configured by default, branch sites gain secure internet access from day one — no manual setup required.
🛡
Enhanced Security
ZIA applies inline threat protection, SSL inspection, and DLP to ensure compliance with Zero Trust security strategies.
↗
Scalable Architecture
The integration scales seamlessly with dynamic branch expansions — no complex per-branch configurations needed.
🔍
Traffic Inspection
Internet traffic is routed securely via IPSec to ZIA for inspection, mitigating the risk of unauthorised or malicious traffic.

Verifying IPSec Status via Console

Run the following command from the appliance shell to confirm IPSec tunnels are properly established:

docker exec strongswan ipsec status
!
INACTIVE TUNNEL
If the IPSec tunnel is inactive or not properly configured, the command will indicate that no active SAs are currently available. This is the first check when troubleshooting connectivity issues between the branch and ZIA.

GRE TUNNEL

Integrating GRE Tunnel

Generic Routing Encapsulation (GRE) is ideal for forwarding internet-bound traffic from your corporate network to the Zscaler service. GRE is a tunneling protocol that encapsulates packets inside a transport protocol — a GRE-capable router encapsulates a payload packet inside a GRE packet, which is then encapsulated in IP.

GRE Key Concepts

PURPOSE
High-Speed Routing
Enables high-speed, secure routing of traffic to ZIA or third-party services with minimal overhead — no encryption processing required.
GRE TUNNELS
Encapsulation
Configured to encapsulate branch forwarding traffic for secure connectivity to cloud or external data centres. GRE encapsulates but does not encrypt.
ZIA INTEGRATION
Nearest GRE Endpoint
Traffic exits through ZIA's nearest GRE endpoints for inline inspection and threat protection — policy is enforced at the ZIA Public Service Edge.
THIRD-PARTY
Partner Routing
Routes traffic to partner networks or applications securely. Supports integration with external data centres and third-party security platforms.
★
KEY BENEFITS
Low latency · Lightweight connectivity · Enhanced scalability for cloud services. GRE tunnels are often built to two ZIA data centres (primary/secondary) to improve availability and failover resiliency.

GRE Traffic Flow

Endpoint → Device Segmentation Gateway → GRE Tunnel(s) → ZIA Cloud → Policy Enforcement → Internet / SaaS

Outcome: Reduced lateral movement (device segmentation) + consistent internet/SaaS protection (ZIA) using GRE tunnel steering. Primary and secondary tunnels provide redundancy and failover.

IPSec vs GRE — When to Use Each

  • IPSec: use when encryption, authentication, and integrity are required — e.g., transmitting confidential or business-critical data
  • GRE: use for cost-effective, lightweight performance tunneling when encryption is not required
  • Both: integrate with ZIA — branch internet traffic is steered through the Zscaler Security Cloud for inspection and policy enforcement regardless of tunnel type
  • Risk with GRE: asymmetric paths or misconfigured steering can cause traffic to bypass ZIA — always verify tunnel configuration

Verifying GRE via Console

ip tunnel

The ip tunnel command lists all GRE tunnel interfaces with their local and remote endpoint details — making it the most direct tool for GRE verification. It is distinct from docker exec strongswan ipsec status (IPSec only) and traceroute (path analysis only).

ZIA LOCATIONS

Understanding ZIA Locations

ZIA Locations identify the various networks from which your organisation sends its internet traffic to Zscaler. When an organisation forwards traffic through a GRE or IPSec tunnel, Zscaler provisions the organisation's IP addresses — which you then add as locations in the ZIA portal. Locations can be added individually or imported via CSV file.

Known vs Unknown traffic: When ZIA receives traffic, it checks whether it originates from a known location (configured in the portal) or an unknown location (remote user traffic). For known locations, ZIA processes the traffic based on location settings — applying authentication, location-based policies, and logging internet activity by location.

Navigation

Infrastructure → Location Management → Locations. The table shows Name, Type, Details, Connection Types, and Sublocations for each configured location.

6 Key Benefits of ZIA Locations

📍
Identify Locations
Identify the locations from which your organisation sends its internet traffic to Zscaler — enabling per-location policy visibility.
🔗
Associate VPN Credentials
Associate a VPN credential, Static IP address, and IPSec/GRE tunnel with a location — linking the tunnel to the correct policy set.
🔒
Create Sub-locations
Create sub-locations for private networks within a larger site — enabling more granular segmentation and policy at the subnet level.
🌐
Associate Virtual Edges
Associate Virtual Service Edges with a specific location — directing branch traffic to the nearest and most appropriate Zscaler PoP.
🚀
Establish Bandwidth Limits
Establish upload and download bandwidth limits per location — enabling QoS and ensuring fair usage across branch sites.
⚙
Enable Features
Enable per-location features like firewall control, authentication requirements, and X-Forwarded-For header reading.
★
OPERATIONAL BENEFITS
ZIA Locations help deliver consistent policy enforcement, better visibility, and easier management across distributed branch environments. Every branch gets the same security posture — enforced by its location configuration, not by local appliances.
