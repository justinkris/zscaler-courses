Deploy and Administer Zero Trust Branch

Site registration, interface roles, console access, and enrolling appliances via Simple Device Activation or Activation Code.

From portal to powered on — Zero Trust from minute one.

Deploying a Zero Trust Branch site isn't just about plugging in hardware. The onboarding workflow ensures every device receives its configuration and policies through a controlled, outbound-managed process — no inbound exposure, no implicit trust, consistent security from the moment the appliance comes online.

MODULE OVERVIEW

What you'll be able to do

This module covers the operational workflow for bringing a Zero Trust Branch site online — from portal registration through device activation. By the end, you'll be able to execute the full deployment process using repeatable, scalable onboarding workflows.

1
Register a Zero Trust Branch site in the admin portal with all required fields
2
Understand interface roles used during deployment — LAN, WAN, Management, and HA
3
Establish basic console-level connection and validate appliance readiness before activation
4
Enroll and activate a Zero Trust Branch appliance using Simple Device Activation or an Activation Code
5
Verify successful activation states — Standalone or Active/Standby for HA deployments
Zero Trust registration principle: A Zero Trust-aligned onboarding approach avoids inbound exposure and implicit trust. Devices onboard via controlled, outbound-managed connectivity so they receive the correct configuration and policies from the start — no inbound ports opened, no manual policy configuration on the device.

REGISTER SITES

Registering a Zero Trust Branch site

Before a physical appliance can be enrolled, a Site must be created in the admin portal. The site is the logical representation of a branch location — it defines the template, DNS, and location binding that the appliance will download on activation.

Navigation path: Infrastructure › Connectors › Edge › Zero Trust Branch › Sites

Site Table Columns

Site Name
The logical name of the branch — used for identification and monitoring. Should follow organisational naming conventions.
Template
The provisioning template applied to the site, defining DNS forwarding and DHCP behaviour (e.g. zt600-ha-lab-custom).
Gateway Name
The name of each physical ZT-series appliance enrolled to this site. HA sites show two gateways per row.
State
Indicates the current HA role of each gateway: Active or Standby.
IP Address
The management or WAN IP address of each gateway appliance.
Version
Firmware version currently running on the appliance (e.g. 8.0, 8P1).

Add Site — Required Fields

Name *
Unique site identifier — e.g. EDU-280-LAB-1. Follow your organisation's naming convention for consistency.
Platform *
Select the appliance model that will be deployed at this site (e.g. zt800).
Template *
Choose the provisioning template (e.g. zt800-standalone-default). Clone the default template via Resources › Templates if a custom version is needed.
WAN DNS Servers *
Specify DNS servers the appliance will use for WAN resolution (e.g. 1.1.1.1, 8.8.8.8). Must be reachable from the branch.
Location *
Select or create the geographic/logical location — links the site to a ZIA location, which automates PSC tunnel creation.
!
CRITICAL DEPENDENCY
Zero Trust Branch must be linked with both ZIA and ZPA in the backend before site creation.

• If ZIA is not linked: the location will not appear under site configuration in the portal
• If ZIA is linked but ZPA is not linked: the strongswan container will not run — IPSec tunnels cannot be established

Required Firewall Ports for Activation

The upstream firewall at the branch must allow the following outbound traffic before activation. Blocked ports are the most common cause of activation failure.

PortProtocolPurpose
UDP 500IPSec IKEKey exchange for IPSec tunnel establishment
UDP 4500IPSec NAT-TIPSec traversal when NAT is present between branch and Zscaler
TCP 443HTTPSManagement channel and control plane communication

Registration Workflow Summary

  • Navigate — Infrastructure › Connectors › Edge › Zero Trust Branch › Sites
  • Create — Click Add Site, fill Name, Platform, Template, WAN DNS, and Location
  • Bind — Link to a ZIA Location to automate PSC tunnel creation
  • Enroll — Use the generated Activation URL or Activation Code (via console)
  • Verify — Confirm gateway states show Active/Standby and site shows Connected

INTERFACES & CONSOLE

Interface roles and console access

Each ZT-series appliance provides dedicated ports for management, LAN, WAN, HA, and console access. Assigning the correct interface roles is essential for secure and reliable branch connectivity.

Port Mapping Reference (ZT-800)

PortInterfaceRole
CConsoleCONSOLE Serial console access (RJ45-to-DB9 or USB)
ge1 / enp2s0f0Port 1WAN 1GbE WAN uplink
ge2 / enp2s0f1Port 2WAN 1GbE WAN uplink (dual ISP)
ge3 / enp2s0f2Port 3MGMT Dedicated management — segregated from data traffic
ge4 / enp2s0f3Port 4LAN / HA* 1GbE LAN or HA sync (HA configs)
ge5–ge8Ports 5–8LAN 1GbE LAN ports for branch switch connectivity

Interface Role Definitions

MANAGEMENT (RED)
Admin & Control
Dedicated port for device management and administrative tasks. Segregated from data-forwarding traffic for improved security and reduced exposure.
LAN (PURPLE)
Branch Internal
Connects the ZT appliance to the branch switch — enabling communication across internal branch devices. Facilitates secure connectivity within the local branch network.
WAN (ORANGE)
Internet Uplink
Connects to the WAN router for internet access. Routes branch traffic to Zscaler PoPs for ZIA inspection and ZPA private app access.

Setting Up Console Access

Console access is required for initial configuration and troubleshooting before the appliance is network-reachable. Use a serial cable (RJ45-to-DB9 or USB) connected to the console port.

★
PUTTY CONSOLE SETTINGS (WINDOWS)
Baud Rate: 115200  ·  Data Bits: 8  ·  Stop Bits: 1  ·  Parity: None  ·  Flow Control: XON/XOFF

Validating Configuration via Console

From the appliance main menu, type 4 to run Show Configuration and verify the appliance settings before activation:

Site Name: FD002-Student-Lab Gateway Name: FD002-student-lab-gw-01 Mgmt IP: not configured WAN Interface: ge1 WAN IP: DHCP (Current: 10.120.1.1/24) WAN Nameservers: 8.8.8.8, 1.1.1.1
!
MANAGEMENT IP NOTE
The management interface IP 192.168.0.99/24 is a temporary address used only during initial setup. It is automatically removed after the appliance is successfully activated — so do not rely on it for ongoing management access.

SIMPLE DEVICE ACTIVATION

Method 1 — Simple Device Activation

Simple Device Activation (SDA) is the preferred enrollment method. It combines centralized templates with a secure onboarding URL, enabling quick and error-free provisioning without any manual device configuration at the branch.

Three Key Features

FEATURE 1 OF 3
Single URL Click Setup
Administrators share a single activation URL with local on-site staff. The setup completes in minutes — no IT expertise required at the branch location.
Reduces time-to-connect from days to minutes.
FEATURE 2 OF 3
Centralised Management
All configurations managed from a single portal. Security teams can monitor, update, and audit every gateway from one pane of glass — regardless of how many sites exist.
Eliminates configuration drift across remote sites.
FEATURE 3 OF 3
Predefined Templates
Templates capture best-practice settings and approved policies. Every new gateway deployment starts from a validated baseline — not from scratch.
Reduces human error and accelerates rollout.

Step-by-Step: Simple Device Activation

1
Validate Configuration
From the appliance main menu, type 4 (Show Configuration) to verify the site name, gateway name, WAN interface, DHCP settings, and DNS nameservers are correct before proceeding.
2
Validate Management Interface
From the ZT shell (Main Menu › Troubleshooting › Shell), run ifconfig or ip addr to confirm the management interface shows IP 192.168.0.99/24. This confirms the appliance is ready for activation.
3
Generate and Share the Activation Link
In the Experience Center portal, navigate to the site and select the gateway. Generate the Activation Link. Share the URL with on-site staff — or email it directly from the portal. On-site staff open the URL on a client machine connected to the appliance management port (192.168.0.x network).
4
Verify Activation Status
The browser shows "Gateway Provisioned" on success. The portal site entry transitions from inactive to "Connecting (1 of 2 tunnels)" then to Connected. Full registration completes within ~15 minutes — network settings are configured automatically.
★
ACTIVATION CONFIRMATION
After successful activation, the portal shows each enrolled gateway's assigned role — Active or Standby — confirming that policy has been downloaded and data tunnels are established. A standalone deployment shows a single gateway in the Active state.

ACTIVATION CODE

Method 2 — Activation Code

The Activation Code is an alternative enrollment method for situations where Simple Device Activation (URL-based) is not practical. It is a unique, time-limited code entered directly into the appliance console — producing the same outcome as SDA.

When to Use Activation Code

  • When remote sites lack browser access or portal connectivity
  • In air-gapped or highly restricted network environments
  • As a fallback when URL-based activation encounters firewall or proxy issues
  • When the activation URL cannot be shared with on-site staff

Key Characteristics

Code Type
Single-use, time-limited code generated from the Experience Center portal alongside the activation URL.
Entry Method
Entered directly via the appliance console interface — no browser or network portal access required.
Outcome
Identical to Simple Device Activation — device receives configuration and policies, tunnels are established, state transitions to Connected.
Typical Completion
~15 minutes from code entry to fully registered and Connected state.
★
TIP
You can email the activation link and code directly from the Experience Center portal to share with remote staff or a managed service provider handling the physical installation. Both the URL and code are available from the same activation dialog.
