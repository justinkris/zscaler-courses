Airgap Protection Modes

Zero Trust Branch offers three protection modes based on the level of isolation required. Each balances security, compatibility, and operational complexity differently.

MODE 1
Airgap
Behavior
Assigns a /32 network mask to each endpoint — "network of one." Devices are isolated and cannot communicate directly on the same subnet. All traffic routes through Device Segmentation gateway.
Use Case
Highly regulated environments requiring complete isolation.
Challenge
May not be supported by all systems due to /32 mask limitations in hardware/software.
MODE 2
Airgap-Lite
Behavior
Uses the subnet mask assigned by the DHCP server — allows endpoints to communicate directly without routing all traffic through the gateway.
Use Case
Networks where /32 is not supported or where full isolation isn't required.
Trade-Off
Less isolation than Airgap but improves compatibility.
MODE 3
Airgap+
Behavior
Supports micro-subnets (/27–/30). Devices in the same micro-subnet can communicate locally; traffic to external destinations or different subnets routes through the gateway.
Use Case
Organisations needing limited local communication alongside secure inspection of external traffic.
Requirements
Device Segmentation gateway must operate in server mode for associated VLANs. Available from Airgap OS 7.7.6+.

ASSETS & TAGS

Managing assets and classifying with tags

Managing Your Assets

The Assets section in the Zero Trust Branch interface gives administrators a clear view of all devices currently connected to the network and their health status. Each asset entry shows:

  • System — OS, Software Rev
  • Security — Protection mode (Airgap/Airgap-Lite/Airgap+), Quarantined status, Risk Score, Secure Posture Score
  • IP Allocation — Assignment Type (DHCP/Static), Last Seen, Lease Interval
  • Tags — automatically assigned tags and any custom tags

Automatic Device Fingerprinting

The system uses fingerprinting techniques to automatically discover and classify devices on the network:

  • DHCP Options — identifies device type from DHCP request parameters
  • HTTP User-Agent — classifies devices based on browser/app signatures
  • SSL Fingerprint — identifies devices through their TLS handshake characteristics

The system also integrates with third-party vendors like Armis to further improve asset discovery accuracy and classification.

Working with Tags

Tags help classify assets based on meaningful attributes like OS, model, brand, or risk level — without relying on IP addresses. This allows flexible, scalable policies that adapt to dynamic environments.

Auto-tagging: Once an asset is discovered, the system automatically assigns tags. If integrated with third-party systems like Armis, additional tags from those systems are also added. Admins can add custom tags to further categorize assets.

Real-World Tags Example

Your organisation uses a NAS device that only works with Windows 7 or Server 2008.

  • Old way: Assign static IPs to those machines and write firewall rules based on IPs — breaks when IPs change
  • Better way with Tags: Create a tag like "Microsoft Windows Kernel 6.1" → Apply it to all Windows 7/Server 2008 devices → Create a policy allowing only devices with that tag to access the NAS. Even if IP addresses change, the policy still works.
★
KEY INSIGHT
Tags aren't just for organisation — they drive segmentation policies. Tag-based policies automatically adapt as devices change IPs, join new subnets, or move between locations. This is what makes Zero Trust Branch dynamically scalable without manual rule updates.

TEMPLATES & OBJECTS

Templates, compatibility, and Objects & Groups

Template Compatibility and Management

Templates provide a standardized way to consolidate configurations for large-scale global deployments of Zero Trust Branch Gateways — ensuring efficiency and consistency across multiple locations.

Navigation: Infrastructure › Connectors › Edge › Zero Trust Branch › Site Templates
DEFAULT TEMPLATES
Prebuilt by System
Non-editable, but clonable. Useful for quick start with minor tweaks. Examples: vm-standalone-default, zt400-ha-default, zt600-enhanced-ha-default, zt800-standalone-default.
CUSTOM TEMPLATES
Created by Admins
Full flexibility — Add, Edit, Delete, Clone. Ideal for large-scale or site-specific deployments. Clone a default template as a starting point, then customise for specific requirements.

Template Compatibility

Templates are available for all platform types: ZT8010, ZT800, ZT600, ZT400, and VM form factors. For each platform, two deployment versions exist:

  • Standalone — for individual gateways (single appliance per site)
  • HA (High Availability) — for redundancy; supports mix-and-match platform types between primary and standby

Why Use Templates?

  • Standardization: Reduces variability across sites — configurations are consistent and auditable
  • Flexibility: Custom templates allow tailored adjustments for site-specific needs without starting from scratch
  • Efficiency: Streamlines deployment, saving time and minimizing errors — scale operations while maintaining reliability

FIREWALL POLICIES & RANSOMWARE KILL SWITCH

Firewall policies and the Ransomware Kill Switch

Zero Trust Branch Firewall Policies

ZTB Firewall policies deliver robust security by isolating and controlling communication between different network environments. Built on a zero trust framework — no connection is trusted by default — effectively preventing lateral movement and minimizing attack surfaces.

Firewall Policy Scope and Evaluation Order

Policies can be applied at four scope levels. Evaluation starts at the most specific (site) and falls back through template, global, then system default. Evaluation stops at the first match — rule order and scope placement directly determine which policy controls traffic.

Ransomware Kill Switch (RKS)

The Ransomware Kill Switch is a one-click attack surface-reduction tool integrated into Zero Trust Branch. It empowers organisations to rapidly contain ransomware threats without disrupting critical business operations.

!
WHY IT MATTERS
Ransomware attacks on IoT/OT environments have surged by 400% year-over-year. 61% of breaches target OT-connected organisations. Standard incident response methods halt critical operations and cause significant business disruption — the RKS solves this with targeted containment.

RKS Core Capabilities

  • One-click Containment: pre-set severity tiers ("virtual fuses" or Defcon levels) for progressive lockdowns — from blocking vulnerable protocols to isolating entire OT segments
  • Maintains Business Uptime: targeted containment without shutdowns — keeps essential operations running
  • Tiered, Policy-based Response: example escalation path: block high-risk access → no non-essential communication → block access to "Crown Jewels" apps
  • Orchestration Integration: fully controllable via APIs for seamless integration with SIEM, SOAR, EDR/XDR tools — enabling automated incident response
  • Integrated Visibility: complete east-west network traffic visibility for all endpoints, real-time granular policy enforcement

RKS Color Codes

Green
Normal operations — maintain normal access and firewall behavior, increase monitoring
Yellow
Elevated caution — restrict some access, heighten alerting
Orange
High risk — enforce Orange mode policies, restrict non-essential comms
Red
Critical containment — enforce Red policies, maximum isolation

ROUTING POLICIES

Branch routing policies and Policy-Based Routing

4 Default Routing Rules

When a new site is added, four default routing policies are automatically created to allow ZIA, ZPA, and Zscaler Cloud Endpoints communication:

  • Rule 1 — Default-ZPA-PBR: Non-ZCC traffic destined to private applications → ZPA
  • Rule 2 — Default-App-Segment-PBR: App Connector app segment discovery of private applications → ZPA (tcp:53, udp:53,67)
  • Rule 3 — Default Zscaler Rule: LAN side traffic destined to Zscaler cloud endpoints → LAN/WAN
  • Rule 4 — Default ZIA Rule: All internet-bound "catch all" traffic → ZIA via IPSec

Policy-Based Routing (PBR)

Zero Trust Branch Gateway aims to replace traditional L2/L3 switches and firewalls. Policy-Based Routing (PBR) enables flexible routing decisions that go beyond standard destination-based routing — steer traffic using intent such as who the traffic is from, what service it's using, or where it's going.

1
Match Traffic
Each packet is inspected against match criteria before deciding how to route it. Unlike traditional routing, PBR can evaluate: source IP address, TCP/UDP ports, destination address.
2
Apply Routing Policy
When a match is found, the associated routing policy takes effect. Policies evaluated in priority order — first match wins. Each policy defines which WAN interface or path the traffic should take and under what conditions.
3
Steer for Outcomes
ZTB dynamically directs traffic across all WAN links: Load Balance (distribute flows across multiple WAN links), Failover (reroute instantly when a link degrades), Optimize (set by priority and link quality).
Configure PBR: Infrastructure › Connectors › Edge › Zero Trust Branch › Sites › Your Site › Routing Policy
Create routing policies: specify Source IP, Destination IP, Primary/Secondary interface, Next Hop, and traffic distribution method.
