VRRP & HIGH AVAILABILITY

Stateful gateway failover with VRRP

VRRP (Virtual Router Redundancy Protocol) is the HA mastership protocol used by Zero Trust Branch. It elects one gateway as Master and another as Backup, providing continuous availability if the Master fails.

Active (Master)
Forwards all traffic. Sends GARP announcements to claim the virtual IP/MAC on the network.
Hot Standby (Backup)
Monitors the Master via heartbeat. Takes over immediately upon Master failure.

Key VRRP Features

  • Non-preemptive failover: Backup does not forcibly reclaim mastership when the original Master returns
  • Conntrack sync: Active session state is mirrored to the Backup for stateful switchover — no dropped sessions
  • Link state monitoring: WAN/uplink health triggers automatic mastership changes
  • GARP: Gratuitous ARP broadcasts update downstream switches after failover
  • Keepalived: The daemon running inside VyOS that manages VRRP timers and advertisements

VRRP Use Cases

USE CASE 1
Continuous Availability
Branch users maintain connectivity even when one gateway goes offline. The Backup takes over the virtual IP without any user-visible interruption.
USE CASE 2
Stateful Switchover
Active sessions are not dropped during failover — Conntrack sync preserves flow state on the Backup before the Master fails.
USE CASE 3
Minimal-Downtime Upgrades
Upgrade one gateway at a time; the other maintains branch connectivity throughout the maintenance window.

HA Troubleshooting Sequence

1
Who is Master?
Identify which gateway currently holds mastership before making any changes.
2
Is the HA path healthy?
Check connectivity on the dedicated HA link (v7.8+) or WAN link (v7.7).
3
Is session state syncing?
Verify Conntrack is replicating flows to the Backup gateway.
4
Did link monitoring trigger switchover?
Review link state events and VRRP logs for automated failover history.

DHCP

Dynamic addressing and device fingerprinting

Zero Trust Branch provides an integrated DHCP server for branch IP addressing across Airgap VLANs, with three modes that balance isolation and compatibility.

DHCP Modes for Airgap VLANs

MODE 1
Airgap
DHCP Server
ON — /32 host masks assigned. Full isolation — each device gets a unique /32 address. All traffic routes through the Device Segmentation gateway.
Use Case
Highly regulated environments requiring complete isolation.
MODE 2
Airgap-Lite
DHCP Server
ON — Original subnet preserved. Devices get addresses from the original VLAN subnet. Less isolation, more compatible.
Use Case
Networks where /32 is not supported or where full isolation isn't required.
MODE 3
Static
DHCP Server
OFF — No DHCP service. Devices use statically configured IP addresses.
Use Case
Environments with existing static addressing that cannot be changed.
Key Concept: Airgap VLANs offer three DHCP modes — full isolation with /32 masks, lite mode preserving original subnets, and static (off) mode — plus CSV-based bulk IP reservations that require DHCP server mode to be active.

IP Reservations

Reserve specific IPs for known devices (servers, printers, cameras) to maintain predictable addressing while still using DHCP. Requirements: ZTB must be the DHCP server for the VLAN, and the VLAN must be created first.

Device Fingerprinting

ZTB uses DHCP fingerprinting to classify devices before full network access is granted — enabling early device identity signals during the DORA handshake.

MAC OUI
First 3 bytes of the MAC identify the device manufacturer (e.g., Apple, Dell, Cisco).
DHCP Fingerprinting
Reads protocol signals during the DORA handshake to classify the device type before an IP is assigned.
DHCP Early Signals
Provides device identity before a full IP is assigned — critical for pre-access classification.
DHCP Options
Option 53 = Message type (VM type). Option 60 = Vendor Class Identifier.

SNMP

Monitoring ZTB gateways with SNMP v2 and v3

Simple Network Management Protocol (SNMP) supports device monitoring by collecting operational status and performance metrics. ZTB supports SNMPv2 and SNMPv3, selected based on security and access-control requirements.

SNMPv2 — Simpler
Community-string based monitoring. Best fit for environments requiring simple setup, efficient scalability, and basic security. No encryption.
SNMPv3 — More Secure
Adds authentication and privacy controls. Best fit for Zero Trust environments prioritizing advanced security, compliance, and user-based access control.

SNMPv2 vs SNMPv3 Comparison

FeatureSNMPv2SNMPv3
Security ProtocolCommunity strings (public/private)HMAC-SHA or HMAC-MD5
AuthenticationCommunity strings onlyUser-based (strong auth)
Data PrivacyPlaintext — no encryptionAES or DES encryption
ScalabilitySmall-to-medium deploymentsLarge, high-security environments
Access ControlBasic network-level controlGranular, user-based permissions

Configure SNMPv2

ZTB supports SNMPv2 via the WAN and Management interfaces. Since SNMPv2 does not encrypt communications, ZTB improves security by allowing only whitelisted SNMP servers to monitor gateways.

Portal path: Infrastructure › Connectors › Edge › Zero Trust Branch › Settings › Global › SNMP Configurations
Enable SNMPv2, set Community value, add Client IPs allowed to poll, and click Save. Download the SNMP MIB for your server.

Validate SNMPv2

Run in the gateway console to confirm the whitelist was applied:

iptables -nvL SECURITY_RULES

Look for ACCEPT rules on UDP port 161 for the whitelisted SNMP manager IPs.

Configure SNMPv3

SNMPv3 adds user authentication and message encryption. Key security capabilities:

Authentication
Ensures messages are exchanged only between verified users, reducing unauthorized access.
Privacy (Encryption)
Uses AES or DES to encrypt SNMP communications — monitoring data cannot be intercepted in transit.
Message Integrity
Prevents tampering by ensuring SNMP messages are not modified in transit.
Access Control
Defines user roles and permissions so only approved users/servers can access or manage the device.
Portal path: Infrastructure › Connectors › Edge › Settings › Global › SNMP Configurations › ⚙ (gear icon)
Build SNMPv3 components: Profile/User, Security Group (noAuthNoPriv / authNoPriv / authPriv), View/OIDs, Trap Parameters.

Verify SNMPv3

Perform an SNMPv3 walk from the SNMP server:

snmpwalk -v3 -l authNoPriv -u admin -a SHA -A "test@123" 10.120.1.5

A successful walk returns MIB OID data from the gateway, confirming SNMPv3 is operational.

Which SNMPv3 feature ensures monitoring data cannot be intercepted or altered while in transit?
✕
Authentication — this verifies user identity, not data confidentiality
✓
Privacy (Encryption) — SNMPv3 Privacy uses AES or DES to encrypt SNMP messages in transit
✕
Access Control — this restricts who can manage the device, not protect data in transit
✕
Message Integrity — this prevents tampering but does not provide encryption

DNS POLICIES

Controlling DNS requests and responses at branch sites

DNS Policies in Zero Trust Branch let you define rules that control DNS requests and responses to your ZTB sites — enabling granular filtering, condition-based actions, and secure routing of DNS traffic.

What DNS Policies Enable

  • Monitor all DNS traffic: Apply policies to all DNS requests and responses regardless of protocol (UDP and TCP)
  • Granular filtering: Use criteria such as source IP/subnet, FQDN, wildcard domains, or any domain
  • Condition-based actions: Allow or reject queries, redirect to specific DNS gateways, or override DNS responses
i
DEFAULT RULE
A system default rule allows and forwards all DNS traffic. It has the lowest precedence and can be overridden by establishing a policy that matches any domain.

Configuration Paths

Site DNS Settings: Infrastructure › Connectors › Edge › Zero Trust Branch › <YOUR-SITE> › Sites › Settings › DNS
Configure WAN DNS Servers (default: 1.1.1.7, 8.8.8.8) and Private DNS Servers.

DNS Policies: Infrastructure › Connectors › Edge › Zero Trust Branch › <YOUR-SITE> › Sites › DNS Policies
Note: All IPv4 in DNS Policies refers to Local Interface IPs (AppC, Private, Airgap, those not directly connected).

Recommended DNS Policy Set — Public DNS Servers (4 Rules)

  • Rule 1: For AppC to resolve FQDN over Central DNS server in the Data Center to their original IP address
  • Rule 2: To resolve applications to their original IP addresses by re-directing specific FQDNs DNS requests over ZPA — IP based Branch to Branch over ZPA
  • Rule 3: To resolve applications to their original IP addresses by re-directing specific FQDNs DNS requests over ZPA — IP based within Branch
  • Rule 4: Private DNS Servers configured with Public Servers. ZCC-ZPA resolves to synthetic IP when Private application FQDN request is sent. Catch-all DNS policy (hidden) handles remaining name resolution using Private DNS servers.

Recommended DNS Policy Set — Private DNS Servers (7 Rules)

  • Rule 1: For AppC to resolve FQDN over Central DNS server in the data center
  • Rule 2: Redirect specific FQDNs DNS requests over ZPA — IP based Branch to Branch over ZPA
  • Rule 3: Redirect specific FQDNs DNS requests over ZPA — IP based within Branch
  • Rule 4: Resolve all domain traffic from ZTB via WAN DNS server directly from WAN interface
  • Rule 5: Resolve Zscaler domain from any users using WAN DNS server
  • Rule 6: Resolve all ZPA application segments to synthetic IP address using ZPA DNS server
  • Rule 7: Default catch-all policy
What is the primary purpose of DNS Policies in Zero Trust Branch?
✕
To convert FQDNs to IP addresses automatically for all users on the internet
✓
To define rules that control DNS requests and responses for ZTB sites, enabling filtering, redirection, and override capabilities
✕
To assign static IP addresses to branch devices and manage DHCP leases
✕
To encrypt all DNS traffic using TLS and ensure DNSSEC validation
