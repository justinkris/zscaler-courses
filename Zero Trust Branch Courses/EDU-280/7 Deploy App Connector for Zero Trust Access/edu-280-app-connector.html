Deploy App Connector for Zero Trust Access

ZPA private access flow, App Connector deployment and validation, BGP/OSPF dynamic routing, and routed tunnels for private site-to-site connectivity.

Private apps, dynamic routing, encrypted tunnels — connectivity without the network exposure.

Zero Trust isn't just about who can access what — it's about how that access happens. App Connectors bring ZPA to the branch without opening inbound ports, BGP keeps routing adaptive, and routed tunnels preserve the source IP visibility that legacy apps depend on.

MODULE OVERVIEW

ZPA private access and dynamic routing in a ZTB environment

This module covers how to extend Zero Trust principles into branch connectivity — enabling secure private application access through ZPA App Connectors, keeping routing dynamic and resilient with BGP/OSPF, and linking branches together with encrypted routed tunnels that preserve source IP.

Learning Objectives

  • Explain the high-level ZPA private access flow and the role of App Connectors
  • Deploy an App Connector and confirm it is running on the gateway
  • Create a ZPA App Segment (FQDN-based) and restrict access by ports as required
  • Configure BGP peering parameters and describe how BGP supports route exchange
  • Configure OSPF in ZTB and verify neighbors/routes using "Show Status"
  • Describe routed tunnel use cases and outline the hub/spoke configuration workflow

Every private application access request in ZPA goes through four stages:

1
User + Client Connector — Initiation
A user at a branch initiates a request to access a private application (ERP, file share, web app). The Zscaler Client Connector intercepts the request and forwards it to the Zero Trust Exchange — without extending the corporate network to the user device. Eliminates the VPN broad-access model.
2
Zero Trust Exchange — Policy Decision
Cloud-delivered policy broker evaluating: Identity-based access (IdP integration), Context-based policy (device posture: OS version, patch level, certificate), and App segmentation (access granted only to the specific application — never the broader network). If conditions are met, a secure micro-tunnel is brokered. If not, access is denied immediately.
3
ZPA App Connector — Secure Tunnel
Lightweight software component deployed close to the private app (DC, private cloud, or public cloud). Key principle: inside-out connectivity — the connector initiates outbound to the Zero Trust Exchange; it never opens inbound ports. No attack surface exposed. The per-session micro-tunnel is scoped to the specific application only and terminated when the session ends.
4
Private Application — Access Granted
User reaches the app but is never placed on the underlying network segment. Access is application-scoped only (no IP-level access, no lateral movement), identity- and policy-bound (revocable in real time if posture changes), and hosted anywhere (DC, AWS, Azure, GCP, or hybrid).
!
DENIED ACCESS
If identity or context does not meet policy, access is denied immediately. App segmentation prevents lateral movement — the user is never placed on the broader network.

APP CONNECTOR DEPLOYMENT

Deploy and validate a ZPA App Connector in a ZTB site

Deploying an App Connector involves three phases: creating the provisioning key in ZPA, defining the App Segments and Access Policy, then integrating the key into the ZTB portal and validating connectivity.

PHASE 1 — ZPA CONSOLE
Create provisioning key + assign connector group
PHASE 2 — ZPA CONSOLE
Create App Segments + Access Policy
PHASE 3 — ZTB PORTAL
Integrate key into site + validate

Phase 1: Create the Provisioning Key

1
Confirm no App Connector exists
Navigate to the ZTB site's ZPA tab to confirm no App Connector is currently integrated.
2
Create provisioning key in ZPA
Private Access > App Connectors > create a new provisioning key. Select the connector certificate, assign a connector group, and choose the deployment location.
3
Copy and save the key
Review settings, save the provisioning key, copy the generated key value, and select Done.
Key Concept: The Provisioning Key is generated in the App Connectors section of the Private Access console. It is used to authenticate and register the connector with the ZPA cloud, then pasted into the ZTB portal to complete the connection.

Phase 2: Configure App Segments & Access Policy

1
Create App Segments
Define the application domain/FQDN (e.g., subnets like 10.20/30) and required protocols and ports. Port examples: RDP = 3389, SSH = 22, Web Server = 80.
2
Select segment & server group
Associate the App Segment with the appropriate segment group and server group.
3
Create an Access Policy rule
Allow the target user group to access the application — specify user group, server group, and the intended connector group.

Phase 3: Integrate & Validate

1
Add connector key to ZTB site
In the ZTB portal: Sites view → select your site → open the ZPA section → add a connector key → paste the provisioning key → save.
2
Confirm Connected status
After a few minutes, the connector status should update to Connected in both the ZTB site view and the Private Access console App Connectors view.
3
Optional: validate on gateway
Run on the gateway console to confirm the App Connector container is running:
docker ps -a

Look for the App Connector container (e.g., pod06Appc) showing a recent start time.

What is the primary purpose of generating a Provisioning Key in the ZPA console during an App Connector deployment?
✕
To set up a link between the segment group, server group, and connector group so traffic is permitted to the application
✓
To generate a unique key that enables the App Connector to register and authenticate with the ZPA cloud during deployment
✕
To define the application domain, specify protocols and port ranges, and assign subnets for RDP, SSH, and web access

BGP & OSPF ROUTING

Dynamic routing for resilient branch networks

BGP and OSPF replace static routing with adaptive path discovery — keeping branches connected, redundant, and scalable as the network evolves.

Border Gateway Protocol (BGP)

  • Dynamic route adjustments: Routing tables update automatically when changes occur — new routes or failures
  • Redundancy: Continuous connectivity by rerouting through alternative paths during WAN link failures
  • Scalable branch management: Supports multiple locations and integrates with cloud services like ZIA
  • BGP with Airgap: ZTB dynamically advertises routes to the network perimeter; static routes are no longer required for onboarded VLANs
BGP Config Path: Infrastructure › Connectors › Edge › Zero Trust Branch › Sites › {YOUR-SITE} › Settings › BGP

Open Shortest Path First (OSPF)

OSPFv2 is a link-state dynamic routing protocol that automatically learns and advertises routes between routers to maintain optimal paths. In ZTB, OSPFv2 can be activated on both the global routing table and the VRF routing table, exchanging routes over both LAN and WAN interfaces.

OSPF Config Path: Infrastructure › Connectors › Edge › Zero Trust Branch › Sites › <YOUR-SITE> › Settings › OSPF › Add

OSPF Configuration Parameters

Area Required
Backbone (Area 0) and Non-Backbone areas. Every OSPF router must belong to at least one area; all non-backbone areas must connect to Area 0.
Network Required
Define networks to advertise to OSPF neighbors. Specifies which interfaces participate in OSPF and which prefixes are in the LSDB.
Router ID Default
Unique 32-bit number per OSPF router. Recommend using the interface IP — avoids conflicts and ensures stable neighbor relationships.
Cost Default
Cost = Reference BW / Interface BW. Lower cost = preferred path. Primary tool for traffic engineering.
Redistribution Optional
Redistribute directly connected routes into OSPF for cross-protocol route sharing and extended reachability.
VRF Optional
Virtual Routing and Forwarding. Isolated routing contexts for multi-tenant or segmented deployments.
Gateway Interface Required
Interface for OSPF Hello packets and neighbor adjacency formation. Select LAN or WAN interface.
Hello Interval Default: 10s
Frequency of Hello packets. Must match on both ends. Reducing enables faster failure detection but increases overhead.
Dead Interval Default: 40s
Time before declaring a neighbor down (4× Hello). Triggers SPF recalculation when exceeded.
Authentication Required
MD5 or SHA keys per interface; must match on both sides. Prevents rogue routers from injecting false routing updates.

OSPF Show Status displays: Interfaces · Neighbors · Database · Routes

ROUTED TUNNELS

Encrypted tunnels with source IP preservation

Routed tunnels provide a secure way to connect branch locations over IP networks while preserving the source IP address required by legacy applications running in branches — such as VoIP, TACACS+, ATMs, and Active FTP.

Branch
(Spoke)
Preserved Source IP + Encrypted · Public IP Network
Hub / DC
(Hub)

How Tunnels Are Established

Automatic Tunnel Creation
ZTB appliance automatically creates routed tunnels toward the hub(s) — no manual configuration required.
Cryptokey Routing
Associates public keys with allowed tunnel IP addresses. Each peer has a private key and public key for authentication, passed via out-of-band method.
Scale Limit
Maximum number of routed tunnels is constrained by the BGP peering scale. Plan tunnel capacity accordingly before deployment.
Source IP Preserved
Tunnel preserves the original client IP — required for legacy applications such as VoIP, TACACS+, ATMs, and Active FTP.

Hub-and-Spoke Configuration

Typical topology: branch spokes (e.g., Pod 30 & Pod 50) connect to Primary and Secondary Hubs over the Internet/WAN, with BGP peering providing redundancy and inter-spoke encrypted connectivity.

Hub Configuration
1. Add On-Prem Hub: Infrastructure › Connectors › Edge › Zero Trust Branch › Hubs
2. Enter Gateway Name & User-Reachable IP (public IP accessible by all spokes)
3. Activate with Generated Key (auto-generated; launches ZTB Gateway)
⚠ Hubs cannot be deployed in public cloud. Incorrect user-reachable IP causes end-to-end failure.
Spoke Configuration
1. Configure Routed Tunnels: Sites › Routing Tunnels › select WAN interface → Connect To Hub
2. Static Routes: steer spoke traffic to hub via WAN interface
3. Share Over All (v8.8.5+): share at VLAN level for BGP advertisement
4. BGP: auto-configures after above steps

Centralized DHCP Over Routed Tunnel

Branch sites can obtain IPs from a centralized DHCP server in the DC. ZTB acts as a DHCP relay — forwarding requests across the routed tunnel using unicast encapsulation.

1
Client Broadcast
Device broadcasts a DHCP Discover message on the local network.
2
Relay Agent
ZTB (configured as DHCP relay) encapsulates the broadcast into a unicast packet with the DC DHCP server as the relay target.
3
Encapsulation & Forwarding
Gateway encapsulates the broadcast and forwards it through the encrypted routed tunnel to the hub, which forwards to the DC DHCP server.
4
Server Response
DC DHCP server processes the unicast request and sends a DHCP Offer back over the routed tunnel to the branch gateway.
5
Delivery to Client
Relay delivers the response, replaces the subnet with /32 (for airgap devices), and delivers the final IP configuration to the client.
In Zero Trust Branch, what does Cryptokey Routing do when a routed tunnel is established?
✕
It encrypts all traffic using symmetric keys generated at the hub and distributed to all spokes
✓
It associates public keys with a list of tunnel IP addresses — each peer uses a private key and public key for authentication, enabling zero-touch secure tunnel formation
✕
It automatically configures BGP peering between all hub and spoke nodes based on discovered IP addresses
