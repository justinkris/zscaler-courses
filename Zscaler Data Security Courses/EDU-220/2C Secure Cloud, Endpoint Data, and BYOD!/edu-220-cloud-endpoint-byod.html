Secure Cloud, Endpoint & BYOD

DSPM for cloud data, Endpoint DLP with user confirmation, dashboard & reports, and browser isolation / PRA for unmanaged devices.

Zscaler secures cloud, endpoint, and BYOD data with robust visibility, risk management, seamless DLP, and secure access.

Whether data lives in AWS, on a corporate laptop, or is accessed from a contractor's personal device — Zscaler provides consistent protection across all three vectors.

◆BEFORE YOU READ — THE BOUNDARY PROBLEM
Inline web traffic is the easy case — a proxy sits between user and internet, and everything passes through. But data lives in places the inline path never sees: a contractor's MacBook at home, an S3 bucket spun up by a developer last week, a PDF dragged from Outlook into a personal Dropbox tab. The same policy must reach all three. Otherwise, the perimeter is just a story we tell ourselves.
  • If your DLP only inspects traffic that crosses the network, what percentage of your sensitive data movement is invisible to it?
  • A cloud bucket, a laptop, and a personal iPad — which of these does your current security stack actually see today?
  • What is the boundary you're trying to protect — the network, the device, or the data itself?
Q1 — MODEL ANSWER
In most modern environments, more than half — sometimes much more. Local USB copies, screenshots, prints, drag-and-drop between apps on the same device, cloud-to-cloud transfers between SaaS apps that never touch your network, AI prompts from personal devices on home wifi — none of it appears in proxy logs. Network-only DLP measures one slice of one channel and calls it coverage. The real number is uncomfortable.
Q2 — MODEL ANSWER
Honestly: usually none of them, fully. The cloud bucket may be visible to a CSPM tool you bought but don't operationalise. The laptop may have an EDR agent but no DLP. The personal iPad accesses everything via SaaS and is completely invisible. The exercise of asking the question is more useful than expecting a yes — it makes the gap concrete and points at which tool fills which seam.
Q3 — MODEL ANSWER
The data itself. The network and device are increasingly impermanent containers — users switch them constantly, and SaaS removes them from the equation entirely. Data-centric security follows the data: classify it, enforce policy at every channel it traverses, and tie access to identity rather than location. The network and device become enforcement surfaces, not the boundary.

ASK A QUESTION
MY NOTES
◆BEFORE YOU READ — SANCTIONED VS SHADOW CLOUD
Cloud storage is the easiest infrastructure in history to provision — a developer with a corporate credit card can spin up an S3 bucket in 90 seconds, and a marketing team can copy a customer database into a "temporary" Snowflake instance before lunch. The data is still yours, the liability is still yours, but the visibility is not. DSPM exists because nobody knows what they have until something asks five specific questions about it.
  • If someone asked you right now where every copy of your customer PII lives across AWS, Azure, and GCP — could you answer in under a week?
  • A bucket created by an engineer last quarter, with public read access, holding a CSV of test data that turned out to be production — how would you discover it before an attacker does?
  • What's the difference between knowing data exists and knowing whether it is exposed?
Q1 — MODEL ANSWER
Without DSPM, almost nobody can. Cloud sprawl, multi-cloud, ephemeral environments, dev/test copies, snapshots, backups, data lakes — sensitive data ends up in dozens of stores nobody is tracking centrally. A week is generous; most organisations cannot answer it in a month, which means in any breach the first painful job is discovery. DSPM exists to make that question answerable in real time, not after the fact.
Q2 — MODEL ANSWER
DSPM continuously scans cloud accounts for stores, classifies their contents, and overlays exposure (public access, over-permissive IAM, shared keys, third-party access). It would flag the bucket as containing PII-class data with public read — exactly the combination an attacker would scan for. Without DSPM, the bucket sits invisible to the security team and visible to the internet, which is the recipe for the cloud breaches that make the news.
Q3 — MODEL ANSWER
Knowing data exists is an inventory question — what is here? Knowing whether it is exposed is a posture question — who can reach it, with what credentials, over which network paths? The first is necessary but not sufficient; the second tells you whether you have a problem. Mature DSPM joins both: a single view that says "this customer PII lives in this bucket and is currently exposed to the internet."
SECURE CLOUD DATA — DSPM

Data Security Posture Management

In today's digital-first world, businesses use cloud platforms like AWS, Azure, and Google Cloud to store, manage, and process data. While the cloud is flexible and scalable, it comes with security risks — data breaches, cyberattacks, and misconfigurations. Protecting sensitive information requires tools that go beyond traditional security measures.

Zscaler enhances cloud security through its zero-trust framework. The Zscaler Data Security Posture Management (DSPM) is a security approach that helps organisations identify, monitor, and protect sensitive data across their environment, ensuring it stays secure and compliant.

How DSPM Works

DSPM deploys two components — a Scanner and an Orchestrator:

  • Scanner — scans data configurations stored at the cloud console; identifies sensitive data (Buckets, VMs, DBs); sends findings to the Orchestrator.
  • Orchestrator — the final step; remediates identified issues through policies, guides, and actionable insights.
  • Metadata stays local — metadata remains within the customer environment and is not transferred outside, preserving data sovereignty.
★
DSPM KEY CAPABILITIES
DSPM provides: Data Discovery (identify sensitive data across cloud), Posture Control (find misconfigurations & vulnerabilities), and Actionable Insights (correlate, prioritise, auto-remediate). It also offers a geographical data breakdown — showing the types and volume of data stored in each location.

ASK A QUESTION
MY NOTES
◆BEFORE YOU READ — THE DEVICE IS ALSO A DESTINATION
A laptop on an airplane has no network. A user dragging a customer list onto a USB stick generates no proxy log. Most DLP conversations focus on data leaving the network — but the device itself is often the final stop. Printing, USB transfer, screenshots, drag-and-drop to a personal Dropbox tab — these are local acts that need local enforcement. And when you do catch the user, the question isn't always "block them" — sometimes it's "ask them why."
  • If your DLP only worked when the user was on the corporate network, what happens at the coffee shop?
  • A user uploads a sensitive file to personal Google Drive — they may be exfiltrating, or they may be working over the weekend. How do you tell the difference without slowing both down?
  • Is a hard block always the right answer, or does a justified soft-block sometimes produce better outcomes — for security AND productivity?
Q1 — MODEL ANSWER
Network-only DLP goes offline with the network. At the coffee shop, the device is on hostile wifi and the user can print, USB-copy, screenshot, and drag-and-drop locally with no inspection. Endpoint DLP — enforced by an agent on the device — keeps policy alive regardless of the network. The agent is the only thing that follows the device home; everything else assumes a network path that may not exist.
Q2 — MODEL ANSWER
Use a soft-block with justification. The user is presented with the policy and asked to enter a business reason; the upload proceeds, the justification is logged, and the security team gets context for review. Legitimate weekend work continues with friction-and-evidence; intent to exfiltrate produces a self-incriminating justification or no justification at all. Soft-block is the cheapest insider-threat signal you can deploy.
Q3 — MODEL ANSWER
No. Hard blocks are right when the action is unambiguously harmful (classified data to a personal cloud, credentials to an attacker's site). Soft blocks with justification are right when intent is the unknown variable — they protect the data, capture the reason, and avoid the productivity damage and shadow-workaround backlash that come from blanket hard blocking. The right answer is the proportional one, and the proportionality lives in the policy.
SECURE ENDPOINT DATA

Endpoint DLP — protecting data on devices

Zscaler's Endpoint Data Loss Prevention (DLP) is a security measure designed to monitor, detect, and protect sensitive or critical data on endpoint devices — laptops, desktops, and mobile phones. It extends monitoring to activities such as printing, saving, uploading, and sharing.

Custom and predefined DLP engines detect sensitive data, allowing or blocking user actions. Endpoint DLP rules are triggered via the Zscaler Client Connector, ensuring incidents are logged even without network connectivity.

User Confirmation Feature

Instead of hard-blocking users, Zscaler offers a User Confirmation (soft block / "Confirm Allow") option within the ZIA Admin Portal. This requires users to justify a sensitive action before proceeding — allowing the operation but capturing their reasoning.

  • Useful for educating users and reducing help desk overhead, especially when testing new rules or dictionaries
  • Justifications come back to the admin as a feedback loop
  • If a user chooses to justify and continue — the operation is allowed (not blocked)
  • User Confirmation severity is not set to high for all printers — it is configurable per use case

Dashboard and Reports

The Endpoint DLP Report (navigate via Zscaler Internet Access → Analytics → Endpoint DLP Report) provides daily insights into end-user activity with sensitive data. It contains two tabs:

  • Overview tab — Activities with Sensitive Data, total Incidents, Users Generating Incidents; Activities Distribution and Incidents Distribution by DLP Engines (PCI, Credit Card, SSN, ChatGPT CatchAll, etc.)
  • Incidents tab — filter by Action Taken (Allow/Block/Exempted/Confirm Allow/Confirm Block), Severity, and Content Type (DLP Dictionaries, DLP Engines, ML Categories)
★
EXAM POINT
The rule name "Block sensitive data to the cloud" is set to protect personal cloud storage — e.g., blocking uploads to personal Google Drive, Dropbox, or OneDrive while allowing uploads to corporate accounts. This is a named rule that appears in the Endpoint DLP policy configuration.

ASK A QUESTION
MY NOTES
◆BEFORE YOU READ — HIDDEN DATA ON A DEVICE YOU DON'T OWN
A contractor logs into your CRM from their personal MacBook. They view a customer record. Now ask: where did that record actually go? Pixels on a screen, sure — but also browser cache, OS-level screenshot service, autofill memory, maybe a synced clipboard on their phone, possibly a backup to their personal iCloud. You did not authorise any of that, and yet all of it just happened. Installing an agent on the device is not an option — they own it, not you.
  • If a user can see data on their screen, what mechanisms exist to prevent that data from being copied, screenshotted, downloaded, or cached locally — when you don't control the device?
  • Traditional VDI solves this by streaming pixels — but at what cost in dollars, latency, and operational complexity? Is there a lighter path?
  • What's the difference between a web app (which fits in a browser) and a thick-client app (which doesn't) — and why does that distinction matter for BYOD security?
Q1 — MODEL ANSWER
Browser Isolation. The application is rendered in a remote, ephemeral browser in Zscaler's cloud and only pixels (and limited input events) are streamed to the BYOD device. Copy/paste, screenshots, downloads, and printing can be selectively disabled per policy. Nothing sensitive ever touches the local browser cache, OS clipboard, or local disk — there is no local copy to leak.
Q2 — MODEL ANSWER
VDI ships a full virtual desktop, which is expensive, latency-sensitive, agent-heavy, and operationally complex — and the user experience is often poor enough that adoption fails. Browser Isolation accomplishes the same data-containment goal for browser-based work without the OS-level overhead. For most BYOD use cases (SaaS, web apps, internal portals), isolation is the right-sized answer; VDI remains for thick-client work that has no browser equivalent.
Q3 — MODEL ANSWER
A web app runs entirely in the browser — its UI, state, and rendering live there — so Browser Isolation can host it remotely and stream pixels. A thick-client app needs a native OS runtime (CAD tools, locally installed Office, custom Windows software) that cannot be containerised in a browser, so for thick clients you fall back to VDI or managed devices. The distinction matters because it tells you which BYOD architecture (isolation vs VDI) is the right fit per app.
SECURE UNMANAGED ASSETS — BYOD

Browser Isolation and Privileged Remote Access

In today's flexible work environment, personal devices are increasingly used for work. While this enhances productivity, it creates vulnerabilities — unmanaged devices can become gateways for data breaches, malware attacks, or unauthorised access to sensitive information.

The solution: Rather than deploying costly VDI infrastructure, Zscaler combines Browser Isolation and Privileged Remote Access (PRA) to provide VDI-like controls for unmanaged devices — without requiring agents or heavy infrastructure.
🌐
Browser Isolation
Manages secure access for widely used web-based applications. Data doesn't land on the unmanaged device — the browser session runs in a cloud container. Covers Web Apps and SaaS apps. Eliminates the need for VDI for most modern work environments.
🔐
Privileged Remote Access (PRA)
Ensures seamless connectivity to internal and privileged systems that require specific protocols (RDP/SSH) without resorting to traditional network-centric models. Covers internal systems requiring specialised access. Agentless.
Deconstructing VDI deployments — Zscaler alternatives: Browser Isolation (Web Apps), PRA (RDP/SSH). Thick client apps remain N/A. Goal: reduce cost and complexity.

Mitigating Data Leakage from BYOD

  • Browser isolation prevents file downloads and restricts clipboard access from unmanaged devices
  • DLP policies can still be enforced on the isolated session — same policies as managed devices
  • ZIA and ZPA direct traffic for SaaS and private apps respectively, even from unmanaged devices
  • The dual approach achieves VDI-like controls in a cost-effective, agile, and scalable manner

MY NOTES
◆BEFORE THE CHECK — TEST YOURSELF FIRST
Before you read the multiple choice — answer in your own words. If a user triggers a soft-block and justifies the action, what happens? Is the "Block sensitive data to the cloud" rule about corporate cloud or personal cloud? Does DSPM tell you only what data exists, or also where in the world it lives? Frame your own answers, then check them against the options.
  • If a user justifies a DLP-triggered print job, is the print allowed or blocked by default?
  • Is the "Block sensitive data to the cloud" rule aimed at corporate Drive/OneDrive accounts, or at personal ones?
  • Does DSPM expose geographical distribution of data, or only the types of data discovered?
Q1 — MODEL ANSWER
Allowed — the entire point of a soft-block plus justification is that the action proceeds once the user accepts policy and gives a reason. The justification is logged for review, but the user is not blocked. If justification were ignored or the action denied anyway, the user would have no incentive to engage with the prompt, and the system would degrade into a hard-block-with-an-extra-click.
Q2 — MODEL ANSWER
Personal cloud accounts. Corporate Drive and OneDrive are the sanctioned destinations — DLP applies its rules there but is not designed to block them. The "Block sensitive data to the cloud" rule targets the exfiltration channel: personal accounts on the same providers (and other unsanctioned cloud storage) that look identical at the domain level but are not your tenant. Tenant Restrictions and content classification together make this distinction enforceable.
Q3 — MODEL ANSWER
Both. DSPM tells you what types of data exist (PII, PCI, source code, secrets) and where — including geographic region, which is essential for regulatory compliance (GDPR, data residency requirements, regional privacy laws). Knowing customer PII exists is useful; knowing it crosses the EU border into a US-region bucket is what determines whether you have a regulatory problem on your hands.
